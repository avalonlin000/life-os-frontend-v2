## 1. 最终目标

从根上重建小雪 LOL 日报生产体系，让日报每天稳定完成：

```text
回顾昨天
→ 明确今天研究哪些比赛
→ 恢复对参赛队伍和选手的完整记忆
→ 补充版本、舆论和盘口
→ 形成逐场赛前判断
→ 用近两个月全量 TK 查漏
→ 验收
→ 发布本地、Wiki、飞书
→ 按项目协议同步和留痕
```

以后增加、删除、调整日报模块，或更换联网来源时，不允许再次打乱正文、漏内容或出现“设计完成但生产未落地”。

---

## 2. 日报核心功能

日报不是一篇让模型自由发挥的分析文章，而是钧钧每天的：

1. 昨日比赛回顾入口。
2. 今日赛前记忆恢复入口。
3. 今日逐场判断入口。
4. TK 防漏保险。

使用比喻：钧钧每天醒来会忘记这些队伍；今天要比赛的队伍就像今天要见面的朋友。日报先帮助钧钧重新认识他们，最后再展示近两个月完整 TK，恢复整体关系图并检查前文是否漏项。

## 3. 根治原则

### 3.1 Markdown 只能是最终成品

必须改成：

```text
数据采集
→ DailyContext 结构化材料包
→ 固定模块卡片
→ 依赖和完整性检查
→ 渲染器一次性生成 Markdown
→ 发布
```

禁止：

- 多个脚本反复修改同一份 Markdown。
- 按标题执行 `replace`、插入或拼接正文。
- 模型直接写整篇日报。
- 基础日报生成后再用全局大块覆盖、重排或补丁式修改。
- 把 Markdown 当数据库、模块通信工具或长期状态。

### 3.2 一个模块必须有明确功能

每个模块必须登记：

- 模块 ID。
- 用户功能。
- 全局模块或单场模块。
- 数据来源。
- 输入字段。
- 依赖模块。
- 固定位置。
- 固定输出格式。
- 缺失处理。
- 完整性指标。
- 是否进入读者正文或只进入 audit。

增加模块走注册；删除模块移除注册；改顺序调整依赖；换联网方式只改采集器；改显示方式只改渲染器。

### 3.3 根治前置：先锁正源，再允许渲染

日报不能把 RAG 搜索结果、文件名猜测或模型判断当成“全量正式知识”。正式上线前必须建立只读的 `daily_knowledge_manifest`，为每条可进入日报的 TK 至少登记：

- 稳定 `tk_id`、原始文件路径、完整原文、日期、实体/主题、来源、内容哈希。
- 知识来源只能是 `knowledge_import`（知识导入）或 `user_confirmed`（钧钧明确确认）。
- 是否可进入日报、是否为压缩/归档派生产物、以及该判定的依据。

旧 TK 先做一次**不改原文的来源审计**：没有上述两种来源证明的记录标为 `legacy_unclassified`，不得被悄悄当作正式知识渲染。`compressed`、归档摘要及其他派生产物不因文件存在就自动进入日报；是否有对应正式原文、是否可展示，必须在 manifest 中可追溯。

参赛队伍 TK 与 60 天总表都只能从 manifest 精确筛选和计数。RAG 仅可辅助模型理解或查找候选，永远不得决定“全量 TK”范围、顺序或数量。

## 4. 唯一读者日报顺序

```text
1. 昨日比赛回顾与认知校准
2. 今日赛程
3. 通用版本理解
4. 今日逐场分析
5. 近两个月全量 TK 总表
6. 只影响用户判断的缺失提示
```

数据来源长表、脚本路径、freshness、模块状态、结构校验和发布日志不得进入读者正文，统一进入 audit。

### 4.1 低级模型可执行的唯一正文骨架

以下标题、层级、顺序和缺失写法是固定合同；渲染器而非模型负责产生标题、表格、分隔和 TK 行。模型只能填自己的“候选校准建议”和“单场判断”卡片字段。

```markdown
# LOL电竞日报 {日期}

## 昨日比赛回顾与认知校准

| ... |

## 今日赛程
| 时间 | 赛事 | 阶段 | 赛制 | 对阵 | 状态 |
|---|---|---|---|---|---|
| ... |

## 通用版本理解
{正式库完整原文，或【通用版本理解缺失】}

## 今日逐场分析
### {队伍A} vs {队伍B}
```

- 今日无白名单比赛时，第二节固定只写 `今日无白名单比赛`，后续“今日逐场分析”固定写 `今日无逐场分析`；版本和 60 天 TK 总表仍照常生成。
- 禁止任意增加、替换或省略上述读者标题；audit 不得混入该骨架。

## 5. 昨日比赛回顾与认知校准

### 5.1 功能

回顾昨天实际发生的比赛，检查已有认识是否得到验证，并提出候选校准建议。

### 5.2 固定内容

1. 昨日赛果：赛事、阶段、赛制、双方、比分。
2. 比赛过程摘要：基于真实赛果和普通联网战报。
3. 原有认知对照：三维、队伍画像、TK 与实际表现的对照。
4. 候选校准建议：建议修改什么、为什么。

### 5.3 知识写入边界

候选建议只展示，不得自动写入 TK、三维、队伍画像或版本理解，也不得作为今天的正式知识直接生效。

正式知识只允许来自：

1. 知识导入系统写入的原始内容。
2. 钧钧明确确认并让小雪记住的内容。

模型分析、赛后推测和校准建议全部标记为 `待确认`。

### 5.4 其他规则

- 赛后战报使用普通搜索、专用来源或本地事实，不使用豆包。
- 上线前必须有独立 `postmatch_collector`，并在来源配置中登记普通搜索或专用战报来源、字段映射、抓取时间、原始链接和失败原因；不得让模型在正文临时联网补写战报。
- 昨天没有比赛就写“昨日无比赛”。
- 不拿更早比赛冒充昨天。
- 不出现独立“分析师视角”板块。

## 6. 今日赛程

### 6.1 功能

只负责告诉钧钧今天研究哪些比赛，不承载分析。

### 6.2 固定字段

- 比赛时间。
- 赛事名称。
- 比赛阶段。
- 赛制。
- 对阵双方。
- 比赛状态。

### 6.3 研究赛事白名单

- 常规时期：只研究 LPL、LCK。
- MSI 期间：只研究 MSI，不附带其他外赛区比赛，也不能把数据库中全部“国际”比赛写进日报。
- 后续杯赛必须明确加入赛事白名单后才能进入日报。
- 必须按赛事名称或赛事 ID 精确过滤，不能只按宽泛 region 过滤。

白名单必须来自一个版本化 `competition_registry`，每个赛事至少包含：赛事 ID（如来源无 ID 则使用规范赛事名）、别名、赛段、启用日期范围、研究状态和来源映射。DailyContext 必须保留：原始候选赛程、命中的白名单规则、被排除的比赛及原因。

- 未命中 registry 的比赛不得靠 `国际`、`INTL`、赛区或模型猜测加入日报。
- registry 缺失、来源赛事名无法消歧、或同一比赛命中冲突规则时，阻止发布而非静默选择。
- MSI 结束、后续杯赛启用或停用只能改 registry 并保留变更历史，不在代码写死日期。

### 6.4 完整性

- 白名单范围内今日比赛一场不能漏。
- 按开赛时间排序。
- 数据库/ScoreGG 筛选后的比赛数必须等于日报比赛数。
- 不一致时禁止发布。

## 7. 通用版本理解

### 7.1 功能

在逐场分析前说明当前比赛环境。

### 7.2 固定内容

- 当前比赛版本。
- 主流体系。
- 体系变种。
- BP 博弈逻辑。
- 版本陷阱。
- 整体版本格局。

### 7.3 展示规则

- 从正式版本理解库完整读取。
- 按正式顺序全文输出。
- 不精选、不摘要、不改写。
- 通用版本环境只在这里展示；单支队伍的版本适应放入对应比赛的定性基本面。
- 缺失或过期明确标注，不由模型临时编造。

### 7.4 可持续修改

```text
钧钧与小雪讨论
→ 小雪整理候选版本理解
→ 钧钧确认
→ 写入正式版本理解库
→ 日报自动读取当前生效完整内容
```

- 未经钧钧确认不得自动修改正式版本理解。
- 支持新增、修改、排序、停用和历史版本。
- 保留版本号、更新时间、来源和生效状态。
- 修改版本理解不需要改日报模板代码。

## 8. 今日逐场分析

每一场比赛必须绑定稳定 `match_id`，并固定按以下顺序：

```text
比赛信息
→ 量化基本面
→ 定性基本面
→ 赛前舆论
→ 盘口与赔率
→ 单场判断
```

任何单场内容不得写入另一场比赛。

### 8.1 比赛信息

- 时间。
- 赛事。
- 阶段。
- 赛制。
- 双方队伍。
- `match_id` 只用于系统关联，可不在读者正文展示。

### 8.2 量化基本面

量化部分只展示客观数据和机械计算，不自由分析。

固定格式：

```markdown
#### 量化基本面

| 指标 | 队伍A | 队伍B | 对比 |
|---|---:|---:|---|
| TS | {值} | {值} | {差值及领先方} |
| mu | {值} | {值} | {差值及领先方} |
| sigma | {值} | {值} | {差值及更稳定方} |
| 近期战绩 | {胜-负} | {胜-负} | {只写客观结果} |
| 数据更新时间 | {时间} | {时间} | — |
```

规则：

- 缺数据统一写 `【缺失】`。
- 对比列只做程序计算。
- 不从数字直接推出交易结论。
- 双方字段完全对称。
- `近期战绩`固定为截至日报生成时、该队最近 5 场**已完赛系列赛**的胜-负；不混入小局、未来赛程或未完赛数据。少于 5 场则照实统计并在 audit 说明。
- `数据更新时间`固定为 TS/mu/sigma 这一来源记录的 `updated_at`，不是日报生成时间；没有可靠更新时间则写 `【缺失】`。

### 8.3 定性基本面

功能：说明这支队伍本身是什么样的队伍。三维、画像、TK 都属于基本面的定性部分，不与“基本面”并列。

每支队伍固定按以下顺序：

```markdown
#### 定性基本面

##### 队伍A

**阵容与选手**
- 上单：
- 打野：
- 中单：
- ADC：
- 辅助：

**队伍三维**

| 维度 | 类型 | 内容 |
|---|---|---|
| 优势局 | {类型} | {原文} |
| 劣势局 | {类型} | {原文} |
| 胜负手 | {类型} | {原文} |

**队伍画像**
- 核心方程：
- 主要赢法：
- 崩溃条件：
- 优势场景：
- 致命弱点：

**队伍版本理解**
{完整原文}

**队伍交易备注**
{全部已确认备注}

**队伍及选手 TK**
{全部正式 TK，最新在前，最近七天加粗}

##### 队伍B

{完全相同结构}
```

规则：

- 队伍和相关选手的全部正式 TK 都必须展示，不精选、不摘要、不改写。
- TK 按时间从新到旧，最近七天整条加粗。
- 缺画像写 `【队伍画像缺失】`，不得跳过整个模块。
- 双方结构完全对称。
- 模型不得生成新的队伍身份描述并冒充正式知识。
- 队伍特征必须描述稳定身份，禁止“碰到强队可能发挥不出来，需要观察”之类无信息量的相对废话。
- 对手造成的具体影响只允许在单场判断中出现，并必须说明具体因果机制。

### 8.4 赛前舆论

功能：展示外界对该场比赛的预期、争议和情绪，不把舆论当比赛事实。

固定格式：

```markdown
#### 赛前舆论

| 来源 | 时间 | 倾向 | 主要观点 |
|---|---|---|---|
| {平台/媒体} | {时间} | {队伍A/队伍B/中立} | {原始观点} |

**主要共识**
- ...

**主要分歧**
- ...

**代表性原话**
- ...
```

豆包搜索策略：

```text
默认统一搜索当天全部白名单比赛 1 次
→ 本地按 match_id、队伍名称和别名拆分
→ 每条结果最多归属一场比赛
→ 无法确认归属的结果不进入正文
→ 检查每场覆盖
→ 缺哪场只补哪场
→ 每日豆包搜索硬上限 3 次
```

这里的“次数”是实际发往豆包/byted 的 HTTP 请求次数，**包括重试**，不是任务数、成功数或 query 数。日报同一日期的 dry-run、publish、失败恢复必须复用第一次冻结的 `online_sources` 资料包，不能因再次运行重复搜索；只有钧钧明确要求刷新才允许创建新采集批次，且仍受当日 3 次总上限。

统一查询必须包含：

```text
{日期} {赛事} 赛前舆论：
{队伍A vs 队伍B}；{队伍C vs 队伍D}
来源：虎扑、微博、贴吧、NGA
```

- 查询构造后必须逐场校验每个白名单对阵的双方规范名或别名均实际出现在 query 中；不得用字符串截断静默丢掉后半场比赛。
- 每条结果必须保存唯一 `request_id`、实际请求序号、归属 `match_id` 和归属依据（队伍名/别名命中）。同时命中多场或无法唯一归属时，归为 `unassigned`，不得进入读者正文。
- 覆盖检查之后才允许补搜；补搜只能针对未覆盖的 `match_id`。预算不足时不再请求，正文如实写缺失，audit 记录未覆盖原因。

规则：

- 豆包/byted 只能用于 `public_opinion`。
- 保留原始 query、标题、URL、站点、摘要、抓取时间和归属依据到 online_sources/audit。
- 正文保留来源和内容时间。
- 新闻、赛程背景不能伪装成社区舆论。
- 没有有效结果写 `【暂无有效赛前舆论】`。
- 不编造比例。
- 豆包搜索预算必须进入自动验收。

### 8.5 盘口与赔率

功能：展示一次采集到的单局市场快照，不追踪盘口变化。

固定格式：

```markdown
#### 盘口与赔率

| 项目 | 盘口 | 赔率 |
|---|---|---|
| 胜负 | {双方} | {双方赔率} |
| 让分 | {盘口线} | {两边赔率} |
| 人头大小 | {盘口线} | {大小赔率} |
| 时间大小 | {分钟线} | {大小赔率} |
```

规则：

- 每场只展示一套单局盘口，不区分第一局、第二局，也不展示初始/当前变化。
- 使用普通搜索、专用盘口来源或以后明确指定的来源，不使用豆包。
- 没有可靠数据写 `【暂无可靠盘口数据】`。
- 不用历史盘口冒充当前盘口。
- 来源和采集时间进入 audit，不占读者正文。
- 上线前必须有独立的 `market_collector`，其来源、字段映射、可靠性条件、失败原因和原始快照均由配置登记；不得把旧 TS 种子赔率、手工备注或模型推断伪装成当日单局盘口。

### 8.6 单场判断

固定格式：

```markdown
#### 单场判断

**队伍A怎么赢**
- {基于画像、三维、TK的具体赢法}

**队伍B怎么赢**
- {具体赢法}

**主要矛盾**
- {双方稳定特征的具体碰撞}

**关键对位与胜负手**
- {具体位置、选手或战术环节}

**比赛预测**
- 倾向：
- 预测比分：
- 主要变数：

**市场对照**
- 基本面与盘口是否一致：
- 市场可能高估什么：
- 市场可能低估什么：

**交易判断**
- 主方向：
- 入场条件：
- 不碰项：
- 主要风险：
```

规则：

- 只能使用本场前文已经展示的量化、画像、三维、TK、版本、舆论和盘口。
- 一涉及模型分析，主要输出定性因果，不重复堆数字。
- 每个结论必须说明具体原因。
- 禁止“中野强势，但碰到强队可能失效，需要继续观察”等空话。
- 不生成新的长期队伍知识。
- 不包含 BP、蓝红方、五英雄或“BP 待确认/复核”。
- 盘口缺失时，“市场对照”固定写 `【暂无可靠盘口数据，不作市场高估/低估判断】`；不得从无盘口材料推导市场结论。

## 9. 近两个月全量 TK 总表

### 9.1 功能

恢复整体记忆，并检查前面逐场内容是否漏掉正式 TK。

### 9.2 范围

- 日报日期向前滚动 60 天，包含当天。
- 范围内所有队伍、选手和主题 TK 全部展示。
- 该 60 天限制只作用于末尾总表，不限制前面参赛队伍“全部相关 TK”的历史范围。

### 9.3 固定格式

```markdown
## 近两个月全量 TK 总表

| 日期 | 实体 | 对象/主题 | TK 内容 | 来源 |
|---|---|---|---|---|
| **最近7天** | **队伍/选手** | **主题** | **完整原文** | **来源** |
| 更早日期 | 队伍/选手 | 主题 | 完整原文 | 来源 |
```

规则：

- 全局按时间从新到旧排列。
- 最近七天整条加粗。
- 不精选、不摘要、不改写。
- 前面比赛区已出现的 TK 在末尾允许并要求重复。
- 60 天窗口拉取数量必须等于表格渲染数量。
- 该表由程序渲染，模型无权删除内容。

### 9.4 全量计数和承载规则

- 每次渲染在 audit 写入 manifest 查询条件、输入 `tk_id` 列表、数量、顺序、内容哈希汇总及渲染数量；四者不一致即阻止发布。
- 当前 60 天全量 TK 原文规模必须在实现前实测，并把真实字符数、各发布端上限和渲染后大小写入验收基线。
- 若本地/Wiki/飞书任一端有单文档上限，必须采用**无损且可核验**的连续附录方案：主日报末尾显示“全量 TK 总表（第 N/M 部分）”，后续附录按顺序紧接，不精选、不摘要、不以链接代替正文；所有分卷共同计数并逐卷验收。未验证承载能力前不得用截断、Top-K、摘要或“详见其他文档”发布。

## 11. 数据与模型契约

### 11.1 DailyContext

DailyContext 至少包含：

- 日期和当前研究赛事白名单。
- 昨日比赛及赛果。
- 今日比赛和稳定 `match_id`。
- TS/mu/sigma/近期成绩。
- 阵容选手。
- 三维。
- 队伍画像。
- 队伍版本理解。
- 通用版本理解。
- 队伍交易备注。
- 队伍及选手全部相关 TK。
- 60 天 TK 窗口。
- 赛前舆论资料包。
- 盘口资料包。
- 缺失项、来源、新鲜度和拉取计数。

### 11.2 模型边界

- 模型只接收局部结构化材料并输出固定 JSON/卡片。
- 程序负责 Markdown 渲染。
- 量化表、正式知识、通用版本和全量 TK 不经过模型改写。
- 模型不得直接操作日报全文。
- 模型输出必须能追溯到输入字段和来源 ID。

### 11.3 读者产物与系统产物分离

读者正文：比赛、记忆、外部信息和判断。

系统产物：

- DailyContext。
- online_sources。
- market_sources。
- 模型卡片。
- audit_report。
- 发布日志。

系统产物不得展开进入读者正文。

### 11.4 版本化、可恢复的生产批次

每一份日报建立稳定 `run_id`，并冻结同一批的 DailyContext、online_sources、market_sources、postmatch_sources、模型卡片和最终 Markdown。批次 manifest 至少保存：

- `run_id`、日期、代码/配置版本、输入文件哈希、最终正文哈希、豆包实际请求数。
- 本地、Wiki、飞书各自的产物 ID/路径、发布时间、回读校验结果和正文哈希。
- 发布失败时的失败目标和原因。

同一 `run_id` 的重跑、发布恢复和飞书重试只能使用冻结输入，不能重新采集联网资料或重新调用豆包。任一发布端失败时保留已成功端的记录，但批次状态只能是 `Verified` 或 `Partial Publish`，不得标 `Released`；恢复时必须回读并确认三端来自同一正文版本。

## 12. 发布前完整性验收

必须自动检查：

1. 今日赛程只包含当前研究赛事白名单。
2. 数据源今日比赛数等于日报比赛数。
3. 每场都有量化基本面、定性基本面、舆论、盘口和单场判断。
4. 两队字段结构对称。
5. 三维、画像、队伍版本和通用版本完整读取；源数据确实不存在时明确缺失。
6. 每支参赛队伍和相关选手的正式 TK 拉取数等于正文渲染数。
7. 60 天 TK 拉取数等于末尾总表渲染数。
8. 最近七天 TK 全部加粗。
9. 舆论结果最多归属一场比赛，不串场。
10. 豆包搜索默认一次、每日最多三次、类别只有 public_opinion。
11. 盘口包含胜负、让分、人头大小和时间大小，或明确写可靠数据缺失。
12. 单场判断位于该场所有材料之后。
13. 正文不存在 BP、独立分析师视角、全局交易大块、工程数据来源长表。
14. 未发生未经确认的 TK、三维、画像或版本写入。
15. Markdown 由单一渲染器生成，没有后加工字符串插入。
16. `competition_registry` 对每场比赛有唯一命中规则；不存在按 region/国际自动推断的漏网比赛。
17. 豆包实际 HTTP 请求总数（含重试、dry-run、publish 和恢复）不超过 3，且同日重跑复用冻结资料包。
18. `daily_knowledge_manifest` 的正式 TK 输入数、渲染数、顺序和哈希对账一致；不存在 `legacy_unclassified` 被当作正式 TK。
19. 盘口和昨日战报分别来自非豆包 collector，或明确记录可靠来源确实无数据；不能由模型或旧数据补造。
20. 三端发布回读的结构、关键正文和内容哈希与同一 `run_id` 一致；未完整发布不得标 `Released`。

处理：

- 系统漏取、漏写、串场、计数不一致、结构错误、知识越权写入：阻止发布。
- 正式数据源本身没有画像、舆论或盘口：允许发布，但正文明确写 `【缺失】`，audit 记录原因。

## 16. 状态定义

只允许以下状态：

- `Ready`：需求已确认，尚未执行。
- `In Progress`：正在实现。
- `Implemented`：代码完成，未完成真实验证。
- `Verified`：本地和影子验证通过。
- `Partial Publish`：本地/Wiki/飞书有部分成功，但未完成同一批次三端回读一致性验证；只能恢复，不能当上线完成。
- `Released`：cron、本地、Wiki、飞书均已真实运行并检查。
- `User Accepted`：钧钧看过正式日报并确认可用。
- `Blocked`：存在必须由钧钧决定或外部状态才能解除的问题。

禁止把设计完成、文档完成、`/tmp` 草稿完成、build 通过或文件存在写成“日报已完成”。
