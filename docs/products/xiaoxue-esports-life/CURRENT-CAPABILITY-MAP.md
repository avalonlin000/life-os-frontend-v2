> 状态：当前事实底图
> 首次核对：2026-07-15
> 最近核对：2026-07-18
> 用途：任何涉及小雪功能、阶段知识整理、TK、版本理解、队伍画像、三维、日报或交易复盘的设计与修改，先读本文件。不要每次重新扫描整个项目。

## 1. 使用规则

本文件回答四个问题：

1. 小雪现在有什么功能。
2. 每项功能真实读取什么。
3. 正式内容最终写到哪里。
4. 哪些入口虽然存在，但当前有冲突、陈旧或尚未走通。

默认直接信任本文件，不重新做全量扫描。只有发生以下任一情况才做局部复查并更新本文件：

- 小雪工作台增加、删除或修改 API。
- 小雪根入口、工具路由或本文件列出的业务 Skill 改名或改路由。
- LOL 数据库相关表结构改变。
- TK、队伍画像、三维或版本理解的正式正源改变。
- 日报生产链、知识导入链或阶段总结链改变。
- 本文件超过 30 天未核对，且任务会正式写入长期知识。

局部复查只检查受影响的行和上下游，不重新扫描无关功能。

## 2. 当前系统结论

小雪是钧钧的个人电竞交易助理，目前主业是 LOL。用户入口固定为：

```text
日报（每日唯一默认入口）
工作台（队伍资料 / 当前赛事 / TK资料库）
小雪对话（动态查询、预案、判断、三层对照、复盘）
后台（数据 / 知识 / 日报 / 巡检 / 备份恢复）
```

当前最重要的事实：

- TK 具有可用的文件库和语义检索，不需要模型逐条扫描全部正文。
- 画像有专门的 `*-team-profile` Skill；Wiki 队伍页是包含画像、三维和队伍 TK 的编译快照，不应反过来成为画像唯一正源。
- 队伍三维正式存放在 `team_3d_data`，属于人工确认内容。
- 日报的通用版本理解读取 `shared_version_analysis`；当前版本 Wiki 已比该表更新，二者存在分叉。队伍旧版本字段已迁移至队伍 TK，不再作为队伍版本栏目。
- 当前 `lol-lineup-analysis` 是纯十英雄完整分析 v8.1：以等经济、正常发育、同等操作水平为前提，先展示版本来源与版本契合，再按24场景筛选、核心链、反制链和四阶段推演，最后给固定32分30秒偏小/偏大/放弃与观赛验证；不使用队伍、选手、TS、三维、当前经济、盘口或赛果。
- `junjun-trading-system` 是明确交易请求的唯一总纲；它可把纯阵容结论作为一项输入，但必须在独立链路核验队伍、市场、趋势、分歧和风险，不能让纯阵容 Skill 承担交易结论。
- 完整阶段知识整理由 `xiaoxue-knowledge-loop` 编排；只更新版本、只更新点名队伍/选手、只压缩 TK 时分别走独立入口。`xiaoxue-knowledge-refresh` 与 `xiaoxue-stage-knowledge` 只保留兼容跳转，旧状态和旧完成结果均不再有效。

## 6. 真实数据流

### 6.1 知识检索与压缩

```text
当前 TK + 归档 TK
  → 元数据粗筛：日期、队伍、选手、来源、精确去重
  → knowledge-rag 语义召回：寻找同义、相反和新增结论
  → 只读取代表性原文做细筛
  → 对照旧版本/旧画像/旧三维形成差分候选
  → 钧钧确认完整候选
  → 写入对应正源并读回
  → 更新 Wiki / RAG / 图谱等镜像
```

向量检索只负责召回，不负责判断哪条结论正确。版本 Wiki 和队伍 Wiki 也只属于镜像/旧基线，不能作为变化证据。每条拟修改或新增结论必须回到阶段内原始 TK 全文，并标出 frontmatter 日期、来源和具体 TK。重复合并、日期整理和候选排序可以自动完成；长期评价、三维变化、分析师分歧和正式写入必须由钧钧确认。

### 6.2 队伍知识

```text
队伍画像 Skill ──┐
三维数据库 ──────┼→ Wiki 队伍编译页 → 工作台展示
队伍/选手 TK ────┘
```

因此，阶段整理不能只读取 Wiki 队伍页。它必须分别读取画像 Skill、三维数据库和 TK；Wiki 只用于核对镜像是否同步。

### 6.3 版本理解

```text
通用版本正式内容 → shared_version_analysis → DailyContext → 日报
                                      └→ 版本 Wiki 镜像

单队版本内容 → team_3d_data.version_understanding → 工作台/Wiki队伍页
```

当前版本 Wiki 与 `shared_version_analysis` 已分叉。下一次正式版本写入必须一次确认后同步两端，并分别读回。

## 7. 已确认的冲突和风险

| 冲突 | 影响 | 后续固定处理 |
|---|---|---|
| 画像 Skill 与 Wiki 编译页不是同一更新节奏 | 工作台可能展示旧画像 | 画像 Skill 作为判断正源；Wiki 作为重建镜像 |
| 通用版本 Wiki 已更新，版本数据库仍是旧版 | 聊天与日报可能使用不同版本理解 | 版本总包确认后同时同步运行正源和镜像 |
| 旧产品文档曾要求复杂阵容长报告 | 同一输入曾得到两种流程 | 2026-07-16 已按 `lol-lineup-analysis` v7.4 收口；后续修改必须保持纯阵容与交易系统分离 |
| 双分析师 API 只读最近 5 条 TK | 无法承担阶段性大批量整理 | 阶段流程使用粗筛/细筛后的小证据集，不直接复用该 API |
| RAG 同时索引当前与归档 TK | 旧结论可能被当成新证据 | 所有命中必须再次按 frontmatter `created` 和阶段范围过滤 |
| 日报由钧钧主动暂停且当天 readiness 缺失 | “流程存在”不等于“今天有日报”，主动暂停也不应显示成系统故障 | 状态表区分能力存在、用户暂停与当日产物存在；未经确认不恢复自动发布 |
| 旧状态以按钮/API 存在证明复盘完成 | 产生“看起来完成但真实路径不通” | 前端无“开始复盘”；完整复盘必须由小雪对话真实走通并验证写回 |
| `/reports/` 当前返回 404 | 工具链接中的历史日报入口不可用 | 不把该链接列为可用入口 |
| 旧阶段 WHAT 加入未经确认的扫描和推翻规则 | 产生额外内容、判断越权且速度慢 | 旧入口与旧状态已隔离；新 `xiaoxue-knowledge-loop` 不继承旧方法 |

## 8. 阶段知识整理的当前实现约束

完整阶段整理的正式方法入口是 `xiaoxue-knowledge-loop`。独立版本更新和独立队伍/选手刷新复用同一套证据、状态与确认合同，但各自在自己的终止态停止；它们不会自动继续到 TK 压缩。单独压缩 TK 仍由 `xiaoxue-tk-compression` 负责。

后续阶段知识整理必须满足：

1. 先提交一份“版本重大差分总表”，一次列完需要改、保持和证据不足的地方；不先处理队伍。
2. 版本确认写入后，系统自动处理全部阶段队伍；关注等级只决定投入顺序和资料容缺，不让钧钧挑批次或逐条回忆材料。
3. 粗筛只做日期、归属、去重、主题聚合、语义召回和候选排序。
4. 细筛只读取每个候选的代表性支持证据、反证和旧结论，不全文扫描所有 TK。
5. 粗筛和细筛属于同一次后台准备；第一条用户可见内容直接是版本重大差分，不展示材料统计，也不询问是否继续。
6. 全部队伍合成一个总整包，只展示发生变化的字段、重要冲突和建议保持不动的关键字段，不逐队确认、不重复整份旧画像。
7. 分析师一致时合并；不一致时展示分歧，小雪不自行裁决。
   只有心语与中年电竞人两个判断都实际生成且一致时，才允许写“无分歧”；缺任一视角不得默认一致。
8. 钧钧的修改先进入临时草稿；重显完整最终版本后才接受确认。
9. 每个变化队伍必须生成至少一条带适用时期和逐条证据的 `实体 | 对象 | 评价` 正文压缩 TK；版本只写通用版本理解，画像 Skill、三维数据库和队伍 TK 分别写回各自正源，未提议字段保持不动。
10. 所有正式写入后必须从真实正源读回，并检查 Wiki/RAG等镜像是否同步。
11. WHAT 不谈仓位、利润和交易 TK；HOW 不修改普通知识、版本、画像和三维。
12. 版本总包只接收跨队伍、可泛化的游戏理解；只在单队成立的 BP、纪律、选手或教练结论进入临时队伍候选，等版本确认后再进整队包。
13. 所有修改、新增和阶段观察在展示前必须通过临时证据账本检查：原始 TK 路径、frontmatter 日期、来源、数量和正文直接支持均合格；失败项删除、降为证据不足或转队伍候选。
14. 每轮先由 `stage_loop.py` 读取带完整性校验的标准状态；只有允许的 phase 可以推进，只有 `version_write_ready` / `teams_write_ready` 可以正式写入。状态目录和布尔凭证不能冒充正式正源，完成命令会自行重读文件或数据库并核对哈希。
15. 用户只需两次确认：版本包用“这版通过”，队伍总整包用“队伍整包通过”。修改意见必须先重显完整包；“继续”不是确认。
16. WHAT 的终止态是 `what_complete`。不存在赛制、赛果、数据整合、存储收尾、能力地图 MSI 标记或最终版图；业务会话不得自改 Skill、SOUL、路由、状态或流程脚本。
