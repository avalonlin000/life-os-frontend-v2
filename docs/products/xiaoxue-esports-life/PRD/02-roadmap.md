# 小雪电竞人生 — Roadmap

> 版本：2.1 | 2026-07-16
> 总纲源：`PRD/04-trading-methodology-and-taxonomy.md`

---

## 落地原则

- 现有能跑通的继续跑通；不能跑通的明确标记，不能伪装完成。
- 每个功能模块单一职责，分表现层、业务逻辑层、数据层。
- 模块间只通过公开接口或事件通信；一个模块失败不得拖垮其他模块。
- 当前只聚焦 LOL，不在日常主路径走通前扩张其他电竞项目。

## P0：定位、职责与窗口统一

目标：统一为“钧钧的个人电竞交易助理，目前主业是 LOL”，并锁定一个主体、三个窗口、一套后台及六项职责。

范围：

1. README / PROJECT_INDEX / BOT_GUIDE 一句话定义统一。
2. PRD/04 作为总纲源进入阅读顺序。
3. PRD/SSD 统一三轴模型：项目轴、使用时间轴、产物对象轴。
4. Wiki 七层入口标注到八层能力。
5. Backlog / Status / Acceptance / Cron 说明同步。

完成标准：

- 文档入口不再把日报或 LOL 横向基本面当作系统总定义。
- 文档明确日报是每日唯一默认入口，但不是小雪的全部能力。
- 文档明确小雪不是自动交易机器人。

---

## P1：恢复并验证真实基础能力

目标：逐项核对队伍、选手、赛程、版本、三维、画像、TK、TS、market_notes 与自动化；能跑通的保留，不能跑通的标记。

范围：

1. 真实读取和写入分别验证，不用接口存在替代真实使用证据。
2. 保留 `market_notes` 主链、TK 正源、队伍交易备注、版本与画像既有边界。
3. 模块故障隔离，不能因单项资料缺失导致其他入口失效。

边界：

- 不自动下交易结论。
- 数据不足写“暂不推荐”。
- 不把 `build_daily_report.py` 改成交易层总线。

---

## P2：走通每日唯一默认入口

目标：日报通过唯一 pipeline 真实生成、展示并明确缺失；每天打开日报即可开始。

范围：

1. 昨日比赛复盘。
2. 当前《交易预案》摘要；没有时明确“尚未制定”。
3. 今日赛程、版本要点、今日逐场《赛前比赛判断》。
4. 最近相关 TK、数据缺失与风险提醒。
5. 只走 `daily_pipeline.py --publish`，禁止旧 postprocess/injector。

边界：

- 日报不分析尚未出现的 BP。
- 日报不机械生成交易对象、入场点或仓位。
- 产物缺失、资料不足和发布失败必须显式显示。

---

## P3：收缩工作台并补齐三层判断

目标：工作台只保留“队伍资料 / 当前赛事 / TK 资料库”，并走通大周期预案、赛前判断、阵容判断、三层对照。

范围：

1. 队伍资料默认全部队伍总览，再进入单队画像、选手、三维、版本理解、TK 和队伍交易备注。
2. 当前赛事展示赛事环境、重点队伍、资料缺口和完整《交易预案》。
3. BP 后阵容判断继续使用纯十英雄 v7.3 固定边界。
4. 小雪对照大周期预案、赛前判断和阵容判断，只输出支持、局部偏离、上层待复核、无法合并。

边界：

- 工作台不放日报副本、BP 输入、聊天、预案生成、三层对照或复盘入口。
- 临场记录隐藏保留，不主动催填。

---

## P4：复盘、长期成长与整体真实验收

目标：完成单场复盘、阶段知识整理、阶段交易复盘、个人交易 TK 和未来提醒，并用真实日常主路径验收。其他电竞项目不在当前范围。

范围：单场复盘、次日日报校准、阶段知识整理、阶段交易复盘、个人交易 TK、相似场景提醒。正式长期知识写入仍需确认并读回验证。

---

## 开发节奏

- 先保证每日主路径真实走通，再扩高级功能。
- 先恢复和验证，再收缩入口，再补齐判断与复盘。
- 任何完成状态都必须有真实页面、API、产物或等价证据。
- 生产代码、cron、日报 publish 和 Wiki 正式日报同步必须单独任务处理，不混入文档总纲改造。
