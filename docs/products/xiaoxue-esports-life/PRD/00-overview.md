## 1. 项目一句话定义

> 小雪是钧钧长期共同发展的个人电竞交易助理；目前主业是 LOL，以完整的 LOL 内容体系为事实和知识基础。

小雪不再被定义为无主体的“系统”、单一工作台或日报工具。产品固定采用“一个主体、三个窗口、一套后台”：小雪是主体；日报、工作台、小雪对话是三个窗口；数据、知识、日报、巡检、备份与恢复是一套后台。

### 六项职责

1. 接住钧钧的自然语言需求并自动分流。
2. 经营完整的 LOL 内容、事实和长期知识。
3. 完成大周期预案、每日准备和赛前比赛判断。
4. BP 后完成独立阵容判断，并完成三层对照。
5. 完成单场复盘、阶段知识整理、阶段交易复盘和经验提醒。
6. 保证数据、日报、知识和自动化真实可靠，异常不得静默。

### 三个窗口

| 窗口 | 唯一职责 |
|---|---|
| 日报 | 每天自动准备好的成品和每日唯一默认入口 |
| 工作台 | 长期 LOL 资料本；主导航固定为队伍资料、当前赛事、TK 资料库 |
| 小雪对话 | 所有动态交流、预案、阵容判断、三层对照和复盘 |

盘口记录继续使用 `market_notes`，但属于隐藏辅助，不进入工作台主导航。

三段用法必须分清：

| 阶段 | 钧钧怎么用 | 小雪核心能力 |
|---|---|---|
| 交易前 | 看今日入口、日报、赛前交易判断日报、队伍资料、TS 表、交易 TK | 建立赛前底稿和不碰项 |
| 交易时 | 蓝红方十英雄出现后先问纯阵容比赛剧本；需要结合队伍、市场和风险时再明确调用交易系统 | `lol-lineup-analysis` v7.4 只看十英雄；`junjun-trading-system` 另行处理交易判断 |
| 交易后 | 补赛果、每局阵容、关键团、盘口实际变化 | 复盘校准并回写 market_notes、队伍 trading_note、TK、三维或画像 |

| 轴 | 分类 | 产品含义 |
|---|---|---|
| 项目轴 | LOL / KPL / DOTA2 / CS2 / Valorant / 其他 | LOL 是第一项目，后续项目先接 schema 和判断链，再做页面 |


## 3. 系统目标

| 序号 | 目标 | 说明 |
|------|------|------|
| T1 | 交易辅助总纲 | 把小雪从 LOL 工作台/日报工具升级为电竞交易辅助系统。 |
| T2 | LOL 第一项目 | 保留并强化 LOL 横向基本面、MSI 环境、队伍/选手、版本、TK、TS 和单场链。 |
| T3 | 交易前入口 | 当前阶段优先做好每日入口、单场赛前底稿、赛前交易判断层和盘口手写草稿。 |
| T4 | 阵容与交易分流 | 蓝红方十英雄 + 比赛版本问题只进 `lol-lineup-analysis` v8.0；明确交易请求进 `junjun-trading-system`，纯阵容结论只作为一项独立输入。 |
| T5 | 交易后校准 | 赛后把判断兑现度、BP 兑现度、市场偏差沉淀回画像、三维、TK、日报或 market_notes。 |
| T6 | 可扩展项目 | 分类必须支持后续接入 KPL / DOTA2 / CS2 / Valorant / 其他。 |
| T7 | 数据可信 | 真实数据库 / Wiki / API / 材料包来源优先；资料不足必须显式标注。 |

## 4. 用户场景

### 核心场景：交易前每日入口

钧钧打开小雪 -> 看日报入口 -> 对每场比赛先看 TS 表、队伍资料、交易 TK、市场分歧和 BP 待补字段。

### 核心场景：LOL 长期依据

钧钧进入基本面 -> 查看 LPL/LCK/MSI/INTL 队伍横向表、队伍画像、选手画像、版本理解、TK 和概念图 -> 形成长期基本面判断。

### 核心场景：纯十英雄阵容分析 v8.0

蓝红方十英雄出来后，钧钧把双方五个英雄直接发给小雪 -> 小雪只触发 `lol-lineup-analysis` -> 在等经济、正常发育、同等操作水平下，先给固定30分钟的偏小/偏大/放弃建议，再输出总体粗略比例、四阶段强度、双方比赛剧本、节奏、人头高/中/低倾向、走势和观赛信号。该链路不读取队伍、选手、TS、三维、当前经济、盘口赔率或赛果，也不产生买卖和仓位建议。若钧钧明确要求“按我的交易系统”，再由 `junjun-trading-system` 独立核验其他资料并消费纯阵容结论。

### 核心场景：交易后复盘校准

比赛结束后，钧钧补比分、每局阵容、关键资源团、盘口实际变化 -> 小雪对齐赛前判断、BP 结构和实际走势 -> 沉淀到画像、三维、TK、日报或 market_notes review。

### 补充场景：跨项目扩展

后续接 KPL / DOTA2 / CS2 / Valorant 时，先定义项目实体 schema、版本/地图/阵容语义、数据源、单场判断链和市场对照层，再接页面和日报。

## 5. 产品边界

| # | 不做什么 | 理由 |
|---|----------|------|
| B1 | 不做自动交易机器人 | 小雪只提供证据、市场分歧、风险和复盘，最终判断由钧钧决定。 |
| B2 | 不把日报当总纲中心 | 日报是交易前入口/发布物，不定义整个系统。 |
| B3 | 不做赛程 App | 赛程只服务分析关联，钧钧看赛程可用外部 App。 |
| B4 | 不把命中率/结算放主流程 | 当前主链路是判断辅助和校准，不是收益率展示。 |
| B5 | 不伪造未出现信息 | BP、首发、赛果、资源团和盘口变化必须等实际出现后再写。 |
| B6 | 不恢复 `tk_library` | 当前知识正源是 Wiki/TK/MemPalace，交易 TK 跟随队伍 TK；旧 RAG 已删除。 |
| B7 | 不恢复旧日报后处理 | 日报只走 `daily_pipeline.py --publish`，旧 postprocess/injector 不进入生产。 |
| B8 | 不和结衣混成一个 Life OS 页面 | 小雪是电竞交易辅助系统，结衣是知行合一系统。 |
