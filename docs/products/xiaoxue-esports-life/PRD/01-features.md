> 版本：2.0 | 2026-07-09

## F1：使用入口层

### 描述

给钧钧、模型和系统提供统一入口，先判断项目、时间段和产物对象，再进入具体能力。当前入口包括 README、PROJECT_INDEX、BOT_GUIDE、工作台顶部页签、今日内容卡和命令栏。

### 输入

- 用户意图：看基本面、看今日内容、写盘口、问 BP、做复盘。
- 项目：当前默认 LOL，后续扩展 KPL / DOTA2 / CS2 / Valorant / 其他。
- 日期：today 或 YYYY-MM-DD。

### 输出

- 正确窗口：日报、工作台、小雪对话；工作台只有队伍资料、当前赛事、TK资料库。
- 正确文档入口：PRD/04 总纲、PRD/03 LOL 第一项目、SSD、Wiki。

### AC

| ACID | 验收条件 |
|---|---|
| F1-1 | README 和 PROJECT_INDEX 一句话定义均为“电竞交易辅助系统，LOL 是第一项目”。 |
| F1-2 | 阅读顺序中 PRD/04 位于 PRD/00 之前或紧随 PROJECT_INDEX。 |
| F1-3 | 日报是每日唯一默认入口；工作台和小雪对话按需使用。 |
| F1-4 | 文档明确小雪不是只查赛程、不是只做日报、不是自动交易机器人。 |

### 当前功能

| 功能 | API / 文档 | 说明 |
|---|---|---|
| 队伍横向表 | `/api/fundamentals/teams?scope=` | 聚合 teams、team_3d_data、Wiki、TK、TS 和资料状态 |
| MSI 环境 | `/api/fundamentals/msi` | 国际赛环境研究，不是赛程表 |
| 队伍画像 | `/api/profile-full/{team}` | Wiki 优先、skill 回退、数据库只读兜底 |
| 队伍三维 | `/api/team-3d/{team}` | 优势局、劣势局、胜负手、战术笔记、版本理解 |

## F3：知识生产与沉淀层

### 描述

把用户记录、B站、公众号、TK、日报、队伍交易备注和赛后复盘沉淀到 Wiki/RAG，可追溯、可检索、可复用。

### 当前功能

| 功能 | 路径 / API | 说明 |
|---|---|---|
| TK 时间资料库 | `/api/tk/library`, `/api/tk/entry/{filename}` | 独立阅读入口；按时间、队伍和关键词浏览，日期可见，正文完整，后台来源信息不进入阅读界面 |
| TK 搜索 | `/api/tk/search` | RAG + Wiki 文件扫描，按日期降序 |
| TK 新增/编辑/删除 | `/api/tk`, `PUT/DELETE /api/tk/{filename}` | 写 Wiki/TK 并触发 reindex |
| 队伍交易备注 | `/api/team-trading-notes`, `/api/team-trading-notes/from-text` | 写回队伍 TK，结构化 `type=trading_note` |
| 知识导入 | `xiaoxue_knowledge_import.py`, `save_to_knowledge_base.py` | B站/公众号/用户资料入 Wiki/RAG |
| 概念图更新 | `update_tk_graph.sh` | 从 TK 生成图谱 |

### AC

| ACID | 验收条件 |
|---|---|
| F3-1 | team_trading_notes 明确归属队伍知识，不新增交易 TK 实体。 |
| F3-2 | 不恢复 `tk_library` 为主链路。 |
| F3-3 | 舆论资料必须先进入材料包，再进入日报正文。 |

## F4：单场判断层

### 描述

围绕一场比赛构建交易前判断底稿，赛前只使用已经存在的信息，不伪造 BP、首发、赛果或赛中走势。

### 当前功能

| 功能 | 路径 / API | 输出 |
|---|---|---|
| 历史分析师入口说明 | `ANALYST-ENTRY-COPY.md` | 历史归档；不得作为当前前端文案或行为正源 |
| 旧单场调用链 | `SINGLE-MATCH-ANALYSIS.md` | 历史兼容资料；不得覆盖当前纯十英雄阵容方法或交易系统边界 |
| TS 单场底表 | `/api/fundamentals/msi-match-context` | mu、sigma、TS、risk_gap、强弱与波动解释 |
| 赛前交易判断日报 | `/api/pre-match-trading-report`, `build_pre_match_trading_report.py` | 主方向/暂不推荐、入场点、市场分歧、不碰项、BP 待确认 |

### AC

| ACID | 验收条件 |
|---|---|
| F4-1 | 单场赛前输出只写双方既有赢法、对局关系、赛前倾向和不确定性，不分析尚未出现的 BP。 |
| F4-2 | 数据不足时输出“暂不推荐”，不硬编方向。 |
| F4-3 | 日报被标注为每日唯一默认入口，但不是小雪的全部能力。 |

## F5：阵容变量层

### 描述

蓝红方十英雄是纯阵容分析的唯一事实输入。队伍、选手、首发身份、ban/pick 过程、TS、三维、当前经济、盘口和赛果属于其他链路，不能混入纯阵容分析。

`lol-lineup-analysis` 是本层唯一正式方法。钧钧给出蓝红方十英雄并询问阵容强弱、阶段变化、比赛剧本、节奏、人头倾向或观赛重点时，只进入该 Skill，不联网、不查数据库、不并行加载其他分析师。

### v8.0 输出合同

| 输出 | 作用 |
|---|---|
| 一句话剧本 | 总体粗略比例 + 最可能的走势；比例不是精确胜率 |
| 强度与流程 | 0–10、10–20、20–30、30 分钟后四阶段谁更舒服、为什么、是否反转 |
| 比赛画像 | 固定30分钟偏小/偏大/放弃建议；节奏快/正常/慢；人头高/中/低倾向；走势；终结与拖延信号冲突时放弃 |
| 高手带看 | 双方各自怎么赢，以及两个最有用的观赛信号 |

### AC

| ACID | 验收条件 |
|---|---|
| F5-1 | 文档明确赛前不得假装已有 BP 或赛后事实。 |
| F5-2 | 分析必须以等经济、正常发育、同等操作水平为默认前提。 |
| F5-3 | 输出不使用队伍、选手、TS、三维、当前经济、盘口、赛果或社区观点。 |
| F5-4 | 不给具体人头数、盘口大小、买卖建议或确定性胜负承诺。 |
| F5-5 | 明确交易请求进入 `junjun-trading-system`；纯阵容结论只可作为其一项独立输入。 |

## F6：市场对照层

### 描述

市场对照层只做赔率、盘口、强弱、波动、入场点、不碰项之间的比较。最终交易判断由钧钧自己定。

### 当前功能

| 功能 | API / 表 | 说明 |
|---|---|---|
| 盘口手写判断 | `/api/market-notes` / `market_notes` | 当前主链路 |
| 旧交易记录 | `/api/trades` / `trade_records` | 兼容层，不回主流程 |
| 队伍交易备注 | `/api/team-trading-notes` | 长期市场观察，跟随队伍 TK |
| 赛前交易判断层 | `/api/pre-match-trading-report` | 交易前发布物，不自动下单 |

### AC

| ACID | 验收条件 |
|---|---|
| F6-1 | `/api/market-notes` 是盘口主链路。 |
| F6-2 | `/api/trades` 标注为历史兼容层。 |
| F6-3 | 不展示命中率/收益率作为主功能。 |
| F6-4 | 不出现自动买、自动卖、自动跟单、必穿、必中等系统目标。 |

## F7：复盘校准层

### 描述

交易后对齐赛前判断、BP 结构和实际走势，校准长期依据和市场备注。

### 输出去向

- 队伍画像：长期结构变化。
- 三维数据：优势局、劣势局、胜负手、版本理解。
- TK：可复用战术/市场/阵容结论。
- 日报/赛后复盘：当日可读沉淀。
- `market_notes.review`：钧钧手写盘口复盘。

### AC

| ACID | 验收条件 |
|---|---|
| F7-1 | 赛后复盘不生成“下次自动买/卖”规则。 |
| F7-2 | 复盘结论必须能归属到画像、三维、TK、日报或 market_notes。 |

## F8：数据工程与自动化层

### 描述

承载数据刷新、DailyContext、冻结资料、单一日报渲染、三端发布回读、健康检查和验收。工程层服务交易辅助系统，但不定义产品总纲。

### 当前功能

| 功能 | 路径 | 边界 |
|---|---|---|
| 数据刷新 | `scoregg_refresh.py` | LOL 数据源专属，后续项目需换数据源 |
| DailyContext | `collect_daily_context.py` | 本地 DB/Wiki 精确 TK manifest/冻结联网材料包；只读 |
| 舆论采集 | `collect_online_sources.py` | 只采 public_opinion，不采 BP/盘口/战报 |
| 单一日报渲染 | `build_daily_report.py` | 从同一 DailyContext 生成唯一读者版，不允许后处理注入 |
| 结构契约 | `daily_report_contract.py` | 顶层顺序、每场顺序、TK/豆包/禁入项对账 |
| 统一发布 | `daily_pipeline.py` | 本地/Wiki hash 回读；飞书主入口+连续分卷回读 |
| 健康检查 | `/api/health`, `xiaoxue_daily_maintenance_report.py` | 真实产物优先，不只看 cron last_status |

### AC

| ACID | 验收条件 |
|---|---|
| F8-1 | 生产、Cron 和 Skill 都只使用 `daily_pipeline.py --publish`。 |
| F8-2 | 日报只从结构化 context 渲染一次，旧 postprocess/injector 不进生产。 |
| F8-3 | 只有本地、Wiki 和飞书全部回读通过时才发布成功。 |
