# 小雪电竞人生 — 系统语义设计文档 (SSD)

> 版本：2.0 | 2026-07-09
> 总纲源：`../PRD/04-trading-methodology-and-taxonomy.md`

---

## 1. 项目一句话定义

> 小雪是钧钧长期共同发展的个人电竞交易助理，目前主业是 LOL。

系统语义先遵守“一个主体、三个窗口、一套后台”。三轴和八层只用于后台分类，不能反向生成第四个用户入口。`lol-lineup-analysis` v7.4 只负责纯十英雄比赛剧本；明确交易请求由 `junjun-trading-system` 负责。

## 2. 用户入口语义

| 对象 | 语义 |
|---|---|
| 小雪 | 唯一主体，负责准备、判断辅助、复盘和长期成长 |
| 日报 | 每日唯一默认入口，展示已加工成品 |
| 工作台 | 长期 LOL 资料本，只有队伍资料、当前赛事、TK 资料库三个主导航 |
| 小雪对话 | 所有动态交流与判断入口 |
| 临场记录 | `market_notes` 隐藏辅助，不是第四个窗口 |
| 后台 | 数据、知识、日报、巡检、备份与恢复 |

---

## 3. 三轴模型

| 轴 | 分类 | 语义 |
|---|---|---|
| 项目轴 | LOL / KPL / DOTA2 / CS2 / Valorant / 其他 | 当前只落 LOL，后续项目按同一模型接入 |
| 使用时间轴 | 交易前 / 交易时 / 交易后 | 赛前证据、临场变量、赛后校准分开 |
| 产物对象轴 | 给钧钧看的 / 给模型用的 / 给系统审计的 | 用户读物、模型上下文、工程审计分开 |

---

## 4. 判断距离

```text
源数据 -> 证据 -> 解释 -> 判断 -> 沉淀 -> 工程
```

| 距离 | 含义 | 当前例子 |
|---|---|---|
| 源数据 | 未解释的事实数据 | schedules、teams、rosters、odds、赛果 |
| 证据 | 可进入判断链的数据 | TS、三维、画像、TK、舆论材料包 |
| 解释 | 为什么这些证据重要 | 强弱、波动、版本适配、资料缺口 |
| 判断 | 面向当前对局的结论 | 既有赢法、对局关系、赛前倾向、不确定性 |
| 沉淀 | 判断后的长期资产 | TK、队伍画像、三维、market_notes review、日报复盘 |
| 工程 | 保证链路可信 | DailyContext、section/slot、hash、health、acceptance |

## 5. 八层能力语义

| 能力层 | 语义 | 代表功能 |
|---|---|---|
| 使用入口层 | 判断从哪里开始 | README、PROJECT_INDEX、今日内容、命令栏、Wiki 00_入口 |
| 长期依据层 | 平时怎么看懂队伍、版本和项目环境 | LOL 横向基本面、MSI 环境、队伍/选手画像、三维、版本理解 |
| 知识生产与沉淀层 | 证据如何进入长期知识库 | TK 搜索/CRUD、概念图、B站/公众号导入、队伍交易备注 |
| 单场判断层 | 一场比赛赛前怎么看 | 分析师入口、TS 单场底表、赛前交易判断日报 |
| 阵容变量层 | 蓝红方十英雄出现后如何推演比赛 | `lol-lineup-analysis` v7.4：总体粗略比例、四阶段强度、比赛剧本、比赛画像、观赛信号 |
| 市场对照层 | 市场是否消化基本面和波动 | market_notes、team_trading_notes、赔率/盘口分歧、不碰项 |
| 复盘校准层 | 赛后如何修正长期依据 | 赛后复盘、画像/三维/TK 回写、market_notes review |
| 数据工程与自动化层 | 数据、发布物和审计如何可信 | DailyContext、日报结构契约、单一 pipeline、health、cron 说明 |

## 6. 核心术语

| 术语 | 定义 | 禁用替代 |
|---|---|---|
| 电竞交易辅助系统 | 组织电竞项目的长期依据、单场判断、市场对照和复盘校准 | 日报工具、自动交易机器人 |
| LOL 第一项目 | 当前最成熟的项目轴实例 | 小雪全部边界 |
| 交易前 | 赛前使用已有证据形成底稿 | 假装已有 BP/赛果 |
| 交易时 | BP、首发、阵容、盘口变化出现后的修正阶段 | 自动下单阶段 |
| `lol-lineup-analysis` | 纯十英雄阵容分析 v7.4；默认等经济、正常发育、同等操作水平，只读本轮十英雄 | 队伍/选手分析、TS/三维、当前经济、盘口/赛果、交易方向 |
| `junjun-trading-system` | 明确交易请求的总纲；核验趋势、稳定性、关注度、分歧、主要矛盾、预案和风险 | 纯阵容分析、自动下注 |
| 交易后 | 赛后复盘并回写长期依据 | 下次自动买卖规则 |
| 横向基本面 | LOL 长期依据层的核心视图 | 总系统定义、赛程表 |
| 日报 | 交易前/交易后发布物之一 | 总纲中心 |
| TK | 可复用的战术、版本、队伍、市场观察知识 | 闲聊灵感 |
| 队伍交易备注 | 跟随队伍 TK 的长期市场观察，`type=trading_note` | 独立交易 TK 实体 |
| market_notes | 钧钧手写盘口判断主链路 | 自动交易记录 |
| `/api/trades` | 历史兼容层 | 当前盘口主链路 |

## 7. 三类产物对象

| 产物对象 | 面向谁 | 代表产物 | 设计原则 |
|---|---|---|---|
| 给钧钧看的 | 钧钧 | 今日内容、日报、赛前交易判断日报、盘口页、队伍横向表 | 可直接读、可手写、少工程噪音 |
| 给模型用的 | 小雪/模型 | DailyContext、TS match context、ANALYST-ENTRY、SINGLE-MATCH、TK 结果 | 结构清晰、字段可追溯、边界明确 |
| 给系统审计的 | 系统/小白 | CRON、health、acceptance、section/slot、hash、marker | 可验证、可定位、可证明未覆盖生产内容 |

## 8. 工作流语义

```text
长期依据层
  -> 交易前单场判断
  -> 市场对照
  -> BP 后 lol-lineup-analysis 纯十英雄比赛剧本
  -> 明确交易请求再进入 junjun-trading-system
  -> 交易后复盘校准
  -> 知识生产与沉淀
  -> 数据工程审计
```

日报只在其中承担发布物职责：

```text
DailyContext -> 单一读者版 -> 语义校验 -> 本地/Wiki/飞书发布回读
```

`build_daily_report.py` 是唯一 Markdown 渲染器；`daily_pipeline.py --publish` 是唯一生产/发布入口，旧后处理链不再进入日报。

纯阵容主入口不是日报或交易链路。钧钧可以直接对小雪说：

```text
小雪，只看十英雄分析这把。
蓝色方：上 / 野 / 中 / 下 / 辅
红色方：上 / 野 / 中 / 下 / 辅
```

小雪必须只路由到 `lol-lineup-analysis`，不把纯阵容请求带入日报、交易系统、搜索或其他分析师。若钧钧明确要求“按我的交易系统”，再独立进入 `junjun-trading-system`。

## 9. 文案原则

- 用“电竞交易辅助系统”，不要把小雪写成单纯日报工具。
- 用“LOL 第一项目”，不要把 LOL 写成系统边界。
- 用“交易前 / 交易时 / 交易后”，不要把所有场景混成日报。
- 用“纯十英雄阵容分析 v7.4 = lol-lineup-analysis”，不要把队伍、选手、TS、三维、当前经济、盘口或赛果塞入该链路。
- 用“明确交易请求 = junjun-trading-system”，纯阵容结论只能作为一项独立输入。
- 用“market_notes 主链路，trades 兼容层”，不要把旧 `/api/trades` 写成盘口主链路。
- 用“队伍交易备注跟随队伍 TK”，不要新增交易 TK 实体。
- 用“暂不推荐 / 资料不足 / 待补”，不要硬编方向。
- 用“钧钧最终判断”，不要写自动买卖、自动跟单、必中、必穿。
