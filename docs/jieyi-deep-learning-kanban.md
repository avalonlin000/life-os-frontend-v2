# 结衣深度学习最终设计与小任务清单

## 1. 目标结论

结衣「深度学习」不是粘贴材料总结器，也不是 NotebookLM 替代品。它的最终定位是：从钧钧已经沉淀在本地 Obs/Wiki 的知识资料库里，围绕一个真实问题自动组织小型学习包，用约 1 小时把零散材料转成钧钧自己的判断、边界和行动。

一句话定义：

> Obs/Wiki 本地资料库 → 结衣选题 → 组织学习包 → 1 小时深挖 → 生成知识卡 / 行动 / 下一问

本阶段只沉淀执行文档，不改功能代码。

## 2. 范围与非目标

### 本次范围

- 明确最终产品交互。
- 明确数据源、模型/API、回流目标。
- 把后续实现拆成可独立验收的小任务。
- 输出本文档：`docs/jieyi-deep-learning-kanban.md`。

### 明确非目标

- 不做功能代码修改。
- 不把入口设计成用户粘贴材料。
- 不做通用 AI 总结工具。
- 不做课程系统或长周期学习管理系统。
- 不把小雪电竞资料混入结衣通用学习产品，除非钧钧明确指定。

## 3. 核心约束

### 3.1 材料源

主材料源是钧钧已经迁移/沉淀的本地 Obs/Wiki 知识资料库。

当前优先路径：

```text
/home/ubuntu/workspace/knowledge/wiki
```

未来兼容环境变量：

```text
OBSIDIAN_VAULT_PATH
```

结衣自己的知/行/思/道记录可以作为辅助上下文，但不能把深度学习窄化成只学习结衣系统笔记。

### 3.2 模型/API

深度学习全流程使用结衣 API，不引入小雪 API，不把项目主责转给小雪或结衣低模型能力。

前端与服务层命名应围绕结衣学习能力，例如：

- `jieyiService.deepLearning.searchCorpus`
- `jieyiService.deepLearning.generateQuestions`
- `jieyiService.deepLearning.createSession`
- `jieyiService.deepLearning.saveResult`

实际实现时可以按现有 shared service 结构调整，但 API 所属语义必须是结衣。

### 3.3 深度标准

深度不是材料长、时间久、问题难，而是完整走过这条链：

```text
问题 → 结构 → 关联旧知识 → 边界/反例 → 判断/行动 → 验收
```

没有边界卡和行动卡，只能算总结，不能算深度学习。

## 4. 最终用户流程

1. 用户点击「开始深度学习」。
2. 用户选择学习范围：最近导入、主题、文件夹、系统推荐、继续上次。
3. 结衣搜索 Obs/Wiki 本地资料库，整理相关材料摘要。
4. 结衣生成 3 个深挖问题，用户选择或轻微改写其中一个。
5. 结衣组织学习包：核心材料、关联旧知识、冲突/边界候选、可迁移场景。
6. 用户进入 1 小时引导学习：问题、结构、关联、边界、行动。
7. 结衣做验收：浅懂 / 半懂 / 可用。
8. 结果按质量回流到 Obs/Wiki、知页草稿、行动项或下一问。

## 5. 页面模块设计

### 5.1 开始区

目的：让用户低成本开始，而不是要求用户先写完美问题。

模块内容：

- 「开始深度学习」主按钮。
- 学习范围选择：最近导入、主题、文件夹、系统推荐、继续上次。
- 粗略主题输入，例如「行动力」「可信度加权」「项目判断」。
- 明确提示：材料来自本地 Obs/Wiki，不需要重新粘贴。

验收标准：用户可以不粘贴材料，仅输入粗略主题就进入下一步。

### 5.2 本地资料搜索与学习包预览

目的：让用户知道结衣找到了什么，并可排除明显不相关材料。

学习包字段：

- 核心材料：最相关的 3–5 条本地资料。
- 关联旧知识：降低理解成本的旧笔记。
- 冲突候选：可能与已有判断矛盾的笔记。
- 边界候选：可能说明该知识不适用的场景。
- 可迁移场景：项目、交易、复盘、执行等可应用场景。

验收标准：学习包必须展示资料来源路径或标题，不能只给一段 AI 生成摘要。

### 5.3 三个深挖问题

目的：结衣替用户从资料里提出值得学的问题，而不是让用户从空白开始。

问题生成要求：

- 每次生成 3 个。
- 每个问题都要附「为什么值得学」。
- 问题必须围绕一个真实判断或行动，不是泛泛总结。

示例：

```text
主题：行动力
A. 我为什么知道该做但还是不动？
B. 什么情况下“缺乏行动力”其实是安全感不足？
C. 怎么把一条知识转成当天能执行的小动作？
```

验收标准：用户可以选择一个问题进入学习会话。

### 5.4 1 小时学习引导

默认节奏：

```text
0-10 分钟：建立全局结构
10-35 分钟：追问关键难点和边界
35-45 分钟：费曼复述
45-55 分钟：验收题 / 反例 / 应用检查
55-60 分钟：回流知/行/思/道，并生成下一问
```

引导卡片：

1. 问题卡：这次到底要弄懂什么。
2. 结构卡：主张、论证链、关键概念。
3. 关联卡：和旧知识的相似、补充、冲突。
4. 边界卡：什么时候不成立，哪里容易误用。
5. 行动卡：它改变接下来什么判断或行为。

验收标准：会话结束时至少产出一句话卡、结构卡、边界卡、行动卡。

### 5.5 结衣验收

验收不是夸奖，而是考试式检查。

检查问题：

- 我能不能不用原文复述？
- 我能不能举自己的例子？
- 我能不能说出一个反例？
- 我能不能说清楚怎么应用？
- 我能不能判断下次该继续学什么？

分级：

- 浅懂：能复述，不能应用。
- 半懂：能解释，有例子，但边界不清。
- 可用：能解释、能举例、能说边界、能转行动。

验收标准：系统必须给出分级，并说明卡在哪一项。

### 5.6 分级回流

回流规则：

```text
浅懂 → 临时笔记 / 下次继续
半懂 → 知页草稿
可用 → 正式知识卡 + 可拆行动
```

Life OS 路由：

- 知：稳定知识、概念、方法。
- 行：立即可执行动作。
- 思：今天的误解、卡点、学习反思。
- 道：长期原则或决策规则。

验收标准：不同验收等级进入不同保存目标，不能一律写成正式知识。

## 6. 后续小任务清单

每个任务都必须可独立验收，避免一次性大任务卡死。

### 任务 1：前端静态入口与状态骨架

范围：只改 `/know` 深度学习模块 UI，不接真实后端。

交付：

- 开始区。
- 范围选择。
- 粗略主题输入。
- “材料来自 Obs/Wiki 本地资料库”的明确文案。
- API 未就绪时的显式占位状态。

验收：

- `pnpm --filter jieyi-web build` 通过。
- `/know` 能看到深度学习入口。
- 页面没有出现“粘贴材料作为主路径”的交互。

### 任务 2：结衣 API 类型与前端服务占位

范围：只加 shared 类型和 service 封装，不要求后端真实返回。

交付：

- DeepLearningSource / DeepLearningQuestion / DeepLearningPack / DeepLearningSession / DeepLearningResult 类型。
- 结衣 deepLearning service 方法。
- 兼容 mock/fallback 数据，明确标注 API 未就绪。

验收：

- `pnpm --filter jieyi-web build` 通过。
- service 调用失败时 UI 显示“API 未就绪 · 使用占位”，不伪装成真实数据。

### 任务 3：Obs/Wiki 本地资料搜索接口

范围：实现后端/本地索引查询接口，只解决“找资料”。

交付：

- 接口读取 `/home/ubuntu/workspace/knowledge/wiki` 或 `OBSIDIAN_VAULT_PATH`。
- 支持按主题/文件夹/最近导入检索。
- 返回标题、路径、摘要、更新时间、相关度。

验收：

- 直接 API 调用能返回真实本地资料路径。
- 空结果时返回可解释空状态，不生成假材料。
- 不读取小雪电竞资料，除非请求参数明确指定。

### 任务 4：三问题生成接口

范围：基于搜索结果生成 3 个深挖问题。

交付：

- 输入：主题 + 候选资料。
- 输出：3 个问题，每个包含问题、理由、关联资料 IDs。
- 使用结衣 API。

验收：

- API 每次返回 3 个问题。
- 问题围绕判断/行动/边界，不是“总结这份材料”。
- 问题能追溯到具体本地资料。

### 任务 5：学习包组装接口

范围：用户选定问题后，组装学习包。

交付：

- 核心材料。
- 关联旧知识。
- 冲突候选。
- 边界候选。
- 可迁移场景。

验收：

- API 返回结构化学习包。
- 每个材料项有来源路径或本地 ID。
- 前端能渲染学习包预览并允许继续。

### 任务 6：1 小时学习会话 UI

范围：实现前端学习会话卡片流。

交付：

- 问题卡、结构卡、关联卡、边界卡、行动卡。
- 用户可逐步填写/确认。
- 会话进度显示。
- 复制/保存按钮占位。

验收：

- `pnpm --filter jieyi-web build` 通过。
- 不依赖真实 API 时仍能用占位数据走完整流程。
- 会话结束前必须出现边界卡和行动卡。

### 任务 7：结衣验收与分级回流

范围：实现验收结果与回流目标选择。

交付：

- 浅懂 / 半懂 / 可用分级。
- 验收问题回答区。
- 回流目标：临时笔记、知页草稿、正式知识卡、行动项、下一问。

验收：

- 不同等级映射到不同回流目标。
- “可用”才允许默认进入正式知识卡。
- 行动项必须能进入行页待执行队列或使用现有 split-to-action 能力。

### 任务 8：真实保存与历史会话

范围：打通保存结果和继续上次。

交付：

- 保存学习结果到 Obs/Wiki 或项目既有知识接口。
- 保存下一问。
- 支持继续上次未完成会话。

验收：

- 保存后能在本地资料库或知识接口读回。
- 刷新页面后历史会话仍可恢复。
- API 失败时不丢失用户已填写内容。

### 任务 9：端到端验收与文档补充

范围：只做验证和补文档。

交付：

- 记录完整链路：主题 → 搜索 → 三问题 → 学习包 → 学习卡 → 验收 → 回流。
- 补充用户说明或当前版本文档。

验收：

- `pnpm --filter jieyi-web build` 通过。
- 至少用一个真实 Obs/Wiki 主题跑通链路。
- 文档说明材料源是 Obs/Wiki 本地资料库、模型/API 是结衣 API。

## 7. 推荐执行顺序

```text
任务 1 → 任务 2 → 任务 3 → 任务 4 → 任务 5 → 任务 6 → 任务 7 → 任务 8 → 任务 9
```

原则：先前端骨架和 API 契约，再接本地资料检索，再做生成和保存。每一步都必须能独立构建或直接 API 验收。

## 8. 验收总标准

最终完成时必须满足：

- 材料主源明确为本地 Obs/Wiki 资料库。
- 模型/API 归属明确为结衣 API。
- 用户不需要粘贴材料即可开始。
- 系统能从本地资料提出 3 个深挖问题。
- 学习过程覆盖结构、关联、边界、行动。
- 结衣验收能区分浅懂 / 半懂 / 可用。
- 回流不是一律正式保存，而是按质量分级。
- 每个实现任务都能独立验收，不依赖一次性大改。
