# 小雪四个独立长流程能力拆分

status: closure revision approved by 钧钧 on 2026-07-18

## Decisions

contract: “只更新版本理解”进入独立版本更新，不自动进入队伍整理或 TK 压缩。

contract: “只更新某队/某选手画像或三维”进入独立队伍知识刷新，范围只覆盖用户指定对象，不强制先更新版本，也不强制压缩普通 TK。

contract: “只导入 B 站视频”与“只导入公众号文章”是两个独立来源执行器；任一来源失败不得阻塞另一来源的候选发现、处理或结果回执。

contract: 原“阶段知识整理”保留为编排入口，顺序调用版本更新、队伍知识刷新和独立 TK 压缩；原“双源知识导入”保留为编排入口，分别调用 B 站与公众号执行器。

invariant: 版本、画像、三维和选手知识仍在原正式正源写入，确认前不写，写后必须读回；拆分不建立第二套业务事实源。版本和画像写入任一步失败必须恢复数据库、文件、凭证和状态。

invariant: 独立队伍/选手确认后只能由固定写入器执行；写入器必须验证确认包、限制文件与数据库字段、失败回滚并生成可复核回执，业务会话不得手写正式目标或凭证。

invariant: 点名选手的刷新状态必须记录选手范围；确认包只能包含该选手的变化与选手画像目标，队伍只作为取证上下文，机器拒绝整队画像、三维、单队版本和其他选手目标。

invariant: 阶段知识整理现有两次确认、证据门禁和 TK 压缩回执保持兼容；独立队伍刷新不因未请求 TK 压缩而制造压缩任务。

invariant: B 站和公众号继续共用同一个确定性知识导入执行器、去重、索引和日志基础设施，不复制两套写入实现。来源发现后自动处理，不要求钧钧确认候选或逐条确认。

invariant: B 站和公众号每个候选必须落入已写入、重复、失败之一；成功项写后读回并保留，失败项可单独重试，全部成功项处理后只更新一次搜索索引。

invariant: 行为包激活必须是整包事务；任一目标激活或读回失败时，已有目标恢复旧内容，新目标删除，并登记回退结果。

failure: 独立来源失败时只报告该来源；双源编排器可以报告部分成功，但不得把一个来源成功冒充双源全部成功。

failure: 独立版本或队伍刷新发生包校验、写入或读回失败时停在原确认阶段，不自动跳到其他模块。

failure: B 站或公众号单条失败不回滚其他成功项；索引失败时明确区分“内容已保存、搜索待补”，下次只补失败或待索引项。

deferred: 本次不修改生产 Cron，不拆日报、纯阵容分析、交易系统或阶段交易复盘。

## User-visible routes

- 版本：只更新版本理解、校准版本知识、看看版本有什么变化。
- 队伍知识：更新 BLG/T1 画像、刷新某队三维、更新某选手画像、补一批参赛队资料。
- B 站：只导入 B 站、补这个 BV、检查并导入新视频。
- 公众号：只导入公众号、补轩窗少年文章、回补公众号遗漏。
- 编排：阶段知识整理仍走完整阶段顺序；导入知识视频仍可一次处理两个来源。

## Acceptance

test: 路由测试证明四类原话各只进入一个独立 Skill，阶段/双源原话仍进入编排器。

test: 状态机测试证明独立版本更新完成或暂缓后直接终止，独立队伍刷新从队伍准备开始并直接终止，原阶段模式仍按版本→队伍→完成推进。

test: 队伍包测试证明独立队伍刷新可在没有普通 TK 压缩请求时完成正式目标读回，阶段模式仍强制压缩覆盖。

test: 独立队伍写入测试证明确认后文件/数据库目标真实写入并读回、失败时全部回滚；点名选手测试证明整队或其他选手目标会被拒绝。

test: 导入测试证明 B 站模式不会调用公众号搜索，公众号模式不会调用 B 站发现，默认双源模式保持兼容并能显式表达部分失败。

test: 自动导入执行器测试证明无需确认即可逐条保存、读回、去重、汇总失败并只更新一次索引；索引失败能保留成功内容并生成可重试回执。

test: 版本写入测试证明完成登记失败时数据库、Wiki 和凭证全部恢复，状态仍可安全重试。

test: 激活测试证明第 N 个目标失败时前 N-1 个目标恢复，新文件删除，登记为回退而非激活完成。

test: 激活后新建真实小雪会话，用标准原话验证第一个 Skill 动作和最重要禁止行为；旧会话历史保留但不得作为验收证据。

test: 激活后从真实 Skill 目录执行完整相关测试；不修改待确认 TK 或 Cron。真实导入验收只处理配置来源中的正常新内容并逐项读回。
