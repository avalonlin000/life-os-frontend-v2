# 结衣精简版 Rill 三层骨架

status: superseded as product root by PRD v4; retained only as infrastructure research
milestone: M1（见 `docs/ROADMAP.md`）
source: `docs/discovery/2026-07-20-jieyi-root-product.md`

external-portfolio: Rill + SwarmVault + Memex；复用 MemPalace；ActivityWatch 延后评估

> 说明：本方案曾用于探索“记录—整理—知识”处理纪律。2026-07-21 起，它不再定义结衣的产品总纲；其中可复用能力只有在服务“认识世界＋改造世界”主循环时才会进入后续设计。

## Goal（历史）

把结衣从“知 / 行 / 思 / 道四条平行能力链”收口为一套以高频记录、复盘、正反馈和知识沉淀为核心的个人进化系统。

## Decisions

contract: 结衣唯一主循环是“原始记录 → 复盘/洞察 → 正式知识 → 可见正反馈 → 继续记录”。

contract: 任意真实输入都可以先原样记录；记录时不要求用户先选择知识、行动、情绪、目标或页面分类。

contract: 外部知识支持两种输入模式：用户主动带入的内容直接进入记录与整理；系统根据现实反馈、重复问题或认知缺口主动发现候选，但必须经过用户选择后才进入同一条验证闭环。

invariant: 原始记录只追加、不覆盖；任何整理、摘要、分类或 AI 处理都必须产生派生结果并保留原始记录。

invariant: 复盘、洞察、趋势和候选均为可重新生成的派生物；不能把派生结果冒充原始事实。

contract: 正式知识是通过用户确认后形成的长期资产，必须保留来源、日期和关联原始记录；重复内容更新已有资产，不无限复制。

contract: 结衣 Agent 负责接收、整理、关联、回看和提出候选；用户保留原始事实纠正、正式知识确认、删除和长期方向改变权。

contract: 正反馈必须让用户看见记录带来的积累，包括新增有效知识、旧记录被重新调用、已验证认识增加或长期轨迹发生变化；不能只显示完成数量或连续天数。

invariant: 主动发现不是无限内容流；每个候选必须关联当前现实问题、待验证判断或已有记录，无法说明用途时不主动推送。

invariant: 情绪、疲劳和状态是影响记录飞轮的保护信号，不成为独立的情绪分析产品，也不能未经判断改写正式知识或长期方向。

contract: 行动、成长领域、阶段目标、当前实践、模式、趋势、每日计划和深度学习都只能作为三层骨架的可选内容或派生视图；它们不得重新成为主入口或新增平行正源。

contract: 旧数据先保留并映射，不在本设计阶段删除、重置或迁移真实数据；任何删除、归档或字段迁移都必须单独验收并可回退。

## Three states

### 1. 原始记录

用户任何时刻产生的真实输入：经历、想法、感受、行动结果、看到的观点、对话片段或问题。

properties: append-only；created_at；原文；可选来源；可选媒体/外部链接；处理状态只记录系统是否已处理，不改变正文。

### 2. 复盘 / 洞察

系统根据原始记录生成的当天回看、周期回顾、关联、变化、候选和建议。

properties: 可重生成；必须带 source_record_ids；明确区分事实、解释、候选；不足时显示数据不足；不得静默写入正式知识。

### 3. 正式知识

用户确认过、可复用、能够影响未来判断或实践的认识。

properties: evergreen；保留来源记录与验证日期；支持更新和合并；支持状态（候选 / 已确认 / 已修订 / 已归档）；删除由用户决定。

## User-visible main path

```text
打开结衣
  → 直接写下或说出任何真实内容
  → 立即得到“已记录”的确认和可选关联
  → 结衣在后台整理，不要求用户现场分类
  → 用户回看当天/本周形成的复盘与正反馈
  → 用户确认其中有价值的内容
  → 进入正式知识，并可在未来被重新调用
```

## Existing capability placement

keep: 知识正文、统一复盘原文、活动事实、认知候选来源、候选提升确认、正式原则内容。

reframe: 行动、成长领域、阶段目标、当前实践、每日计划、深度学习、模式/阻力/趋势；先改为记录类型、复盘视图或知识验证上下文，不继续扩展独立入口。

archive-candidate: 当前四页导航、任务队列、独立目标/智慧池、复杂候选池和多套 Cron；只有完成三层主路径验证后才决定保留、合并或归档。

## Failure behavior

failure: 记录保存失败时必须明确告诉用户未保存，不产生“已记录”的假反馈。

failure: 整理失败不影响原始记录；页面展示“原始记录已保留，整理稍后可重试”。

failure: 资料不足时不生成模式、原则或积极结论；只显示事实和不足范围。

failure: 正式知识确认或写回失败时保持候选状态，不显示已沉淀。

## Acceptance

test: 用户只输入一段自然语言，不选择页面类型，也能保存为一条原始记录，刷新后仍能读回原文。

test: 同一原始记录重复触发整理不会重复创建知识，且不会修改原始记录。

test: 复盘结果能显示来源记录；无记录或数据不足时明确显示不足，不生成伪洞察。

test: 用户确认候选后才能形成正式知识；正式知识可回溯到原始记录，重复确认保持幂等。

test: 原始记录保存成功但整理失败时，用户仍可查看和重试，不丢失原文。

test: 至少一条真实记录从输入、回看、候选确认到正式知识读回走通；旧行动/计划数据仍保持可恢复。

## Out of scope

- 本设计不决定最终数据库、Markdown、MCP 或具体模型实现。
- 本设计不立即删除现有页面、表、Cron、知识文件或 Agent Skill。
- 本设计不把情绪诊断、连续打卡、目标管理或任务管理重新包装成主产品。

## Working notes

- Rill 的 append-only input、可重跑 distill、来源追踪和用户确认门是主要参考；Rill 的 projects/tasks/workspaces/newsletter/pulse 体系不作为结衣默认结构。
- 结衣现有产品库中派生行动和每日整理数量高于原始活动/复盘输入，说明第一阶段必须先恢复记录入口，而不是继续补分析模块。
- 正反馈是产品结果，不单独建一套“积分/连续天数”数据源。
