# 钧钧的 Vibe Coding 使用手册

> 版本：1.0
>
> 日期：2026-07-10
>
> 适用范围：Life OS 母工程、结衣知行合一、小雪电竞人生，以及相关共享工程和运行维护。
>
> 目标：让钧钧只需要说清“想得到什么”，同时让每次修改都能被复现、验证、回滚和继续推进。

## 1. 先记住一句话

```text
一个 Goal 管一个结果；一个 Change ID 串起需求、修改、验证、记录和 Git；没有真实证据就不叫完成。
```

Vibe Coding 不是随口说一句、让模型不停改。好的 Vibe Coding 是：钧钧负责目标、体验和取舍，Codex 负责归类、实现、验证和交付，文档与 Git 负责让下一次还能接着做。

## 2. 谁负责什么

| 角色 | 负责 | 不负责 |
|---|---|---|
| 钧钧 | 提 Goal、体验要求、优先级和最终取舍 | 记路径、补同步、操作 Git、猜技术细节 |
| Codex | 长期默认开发主力：归类、方案、实现、验证、必要同步和交付 | 未授权的破坏性或系统级动作 |
| 小白 | 飞书入口、Hermes 运维、bot 恢复和 Codex 不可用时备用接手 | 日常开发主控和重复形式复验 |
| 结衣 | 知行合一业务和日常使用辅助 | 管理工程项目 |
| 小雪 | 电竞交易辅助业务和日常使用辅助 | 管理工程项目或自动交易 |
| 文档与 Git | 分别保存目标/契约、证据和版本点 | 代替真实运行验证 |

钧钧始终可以直接把 Goal 给 Codex，这是一条长期关系，不随项目阶段切换。小白只有在 Codex 不可用、需要 Hermes 运维或钧钧明确指定时接手；无论由谁接手，本文的 Change ID、状态和验收规则都不变。

## 3. 钧钧平时怎么说

### 3.1 最短 Goal 模板

大多数任务只需要五句话：

```text
Goal：我想得到什么结果。
现状：现在最不满意或出错的地方是什么。
保持：哪些现有体验、数据或入口不能变。
验收：我看到什么就算成功。
边界：哪些事情这次不要做，或哪些动作必须先问我。
```

示例：

```text
Goal：让小雪在 BP 出来后能快速生成阵容分析提示，并能保存本场复盘入口。
现状：现在信息散在分析师和交易区，我每次都要重新组织文字。
保持：现有赛前日报、盘口草稿和 TK 搜索不能受影响。
验收：我填完蓝红方和阵容后，可以一键复制八步法提示；保存后重新打开仍能看到。
边界：不自动下单，不改生产 cron，不伪造缺失的 BP 数据。
```

如果钧钧只说了 Goal，执行者应从仓库补齐技术范围；只有产品取舍、破坏性数据、系统配置或安全边界不明确时，才需要回来确认。

### 3.2 三类常用指令

#### 新功能

```text
给我增加【用户行为】。
入口在【页面/对话/流程】。
成功时看到【结果】，没数据时看到【空态】，失败时看到【错误态】。
不能影响【旧功能】。
完成后按真实运行路径验收并留修改记录。
```

#### 修改已有功能

```text
把【当前行为】改成【目标行为】。
先查清它现在来自前端、API、数据库还是文档，不要只改表面文字。
保持【兼容项】。
用【具体场景】验收，给出改前/改后和回滚点。
```

#### 全面复盘

```text
只读复盘【范围】的 Git、工作树、运行态、PRD/SSD、Backlog 和 Delivery。
输出时间线、反复痛点、根因、证据和优先级。
不要直接改代码；先给我可执行的下一组 Goal。
```

## 4. 每次修改的标准流程

### 第 0 步：建立 Change ID

格式：`VC-YYYYMMDD-NN`，例如 `VC-20260710-01`。

同一个结果即使横跨母仓库和独立小雪仓库，也使用同一个 Change ID。不同结果不能因为“顺手”塞进同一个 Change ID。

### 第 1 步：归类

| 需求关键词 | 归属 | 主代码位置 |
|---|---|---|
| 知、行、思、道、复盘、行动、原则 | 结衣知行合一 | `/home/ubuntu/life-os-frontend-v2/packages/jieyi-web/` |
| 电竞、LOL、队伍、TK、TS、盘口、BP、日报 | 小雪电竞人生 | `/home/ubuntu/xiaoxue-web/` |
| Delivery、索引、bot 边界、共享脚本 | 共享工程层 | `/home/ubuntu/life-os-frontend-v2/` |
| gateway、模型、bridge、系统服务 | Hermes/系统运维 | `/home/ubuntu/.hermes/` 或系统配置 |

如果同时命中两个产品，应先判断是一个真正的跨产品能力，还是两个独立 Goal。默认拆开。

### 第 2 步：记录基线

开始改动前必须记录：

- 当前分支、HEAD 和是否领先远端。
- 工作树是否已有改动，哪些属于别人或旧任务。
- 当前运行入口是否能复现问题。
- 本次必须读的目标文件和真实代码入口。
- 是否涉及数据库、生产服务、cron、密钥、网络或外部消息。

最小命令：

```bash
git -C /home/ubuntu/life-os-frontend-v2 status --short --branch
git -C /home/ubuntu/life-os-frontend-v2 log -5 --oneline
git -C /home/ubuntu/xiaoxue-web status --short --branch
git -C /home/ubuntu/xiaoxue-web log -5 --oneline
```

规则：已有不明改动不覆盖、不重置、不混入本次提交。需要在同一文件继续修改时，先说明重叠范围。

### 第 3 步：写 Goal Card

执行者把自然语言 Goal 结构化为：

```text
Change ID：
目标结果：
项目归属：
用户入口：
现状证据：
允许修改：
禁止修改：
验收场景：
回归范围：
风险/需确认项：
事实源与需要同步的位置：
```

Goal Card 是本次修改的边界，不是新的产品总纲。实施中发现新需求，放入 Inbox 或新 Goal，不静默扩张当前范围。

### 第 4 步：拆成可验证切片

一个好切片通常是：

- 一个用户可感知行为；或
- 一个前后端契约变化；或
- 一个独立运维/文档治理结果。

一个功能切片应一次包含必要的成功态、空态、错误态和兼容检查。不要把同一行为机械拆成“入口、提示、错误态、日期、文档”五次交付。

建议一个 Goal 有 1 至 5 个切片。每个切片完成后都能独立验证；未完成切片不得用文档描述伪装成已实现。

### 第 5 步：先查真实链路，再改

按这个顺序找根因：

```text
用户看到的页面/对话
-> 前端状态和请求
-> API 路由与响应
-> 服务层/脚本
-> 数据库或知识来源
-> 运行服务和代理
```

只改文案不能修复数据链路；只改前端 fallback 不能证明后端真实可用；只写 PRD 不能证明功能已经上线。

### 第 6 步：实施时控制变更面

- 优先修改事实源，其他地方引用或写短摘要。
- 不为了“所有文档看起来一致”机械复制同一段话。
- 不修改与 Goal 无关的文件。
- 发现现有实现与目标冲突时，先报告冲突和证据，再做产品取舍。
- 跨仓库改动使用同一个 Change ID，但分别构建、分别提交。
- 破坏性动作、真实业务数据写入、生产 cron、密钥/模型/账号、系统网络必须先确认。

### 第 7 步：按四层验收

| 层 | 要证明什么 | 常见证据 |
|---|---|---|
| 静态/构建 | 代码能编译，语法和类型不过期 | build、typecheck、py_compile、node --check |
| 契约/API | 请求、响应、错误和写回符合约定 | curl、响应片段、数据库只读核对 |
| 用户路径 | 用户从真实入口完成目标行为 | 浏览器步骤、截图、页面状态、重开读回 |
| 回归 | 原有关键能力没有被破坏 | 受影响路由/API 的最小回归矩阵 |

当前最低验证：

```bash
# 结衣
cd /home/ubuntu/life-os-frontend-v2
pnpm build:jieyi
curl http://127.0.0.1:3001/
curl http://127.0.0.1:3001/know
curl http://127.0.0.1:3001/act
curl http://127.0.0.1:3001/reflect
curl http://127.0.0.1:3001/way

# 小雪
cd /home/ubuntu/xiaoxue-web
npm run build
python3 -m py_compile main.py
curl http://127.0.0.1:8880/
curl http://127.0.0.1:8880/api/teams
```

只跑 build 只能证明“能构建”，不能证明 API、数据和交互正确。文档任务至少要做链接、关键词、diff 和事实源一致性检查。

### 第 8 步：使用统一状态

| 状态 | 含义 | 可以怎么说 |
|---|---|---|
| Planned | 已决定，但未实现 | 已列入计划 |
| In Progress | 正在修改，尚未完成验证 | 正在推进 |
| Implemented | 已写入代码/文档，但验证不完整 | 已实现，待验证 |
| Verified | 约定验收已通过，有证据 | 已验证 |
| Released | 已进入目标运行环境并做发布后 smoke | 已上线/已发布 |
| Blocked | 明确阻塞且无法继续 | 写清阻塞证据和解除条件 |
| Abandoned | 明确放弃或被替代 | 写清替代方案 |

禁止把“方案写完”“文档同步”“代码写完”“本地构建通过”都简称为 Done。产品整体完成度必须按 Backlog/Acceptance 计算，不能从单个 Delivery 推断。

### 第 9 步：只同步受影响的事实源

| 变化类型 | 必须更新 | 通常不用更新 |
|---|---|---|
| 产品定位/边界 | 总纲 PRD、索引、必要的 Bot Guide | 每一份 SSD 重复整段总纲 |
| 用户功能 | Features/Backlog/Acceptance、对应代码 | team 运维说明 |
| API/schema | SSD data/API、类型/服务、验收 | 产品总览全文 |
| UI 主路径 | UI spec、页面代码、用户说明 | 数据方法论文档 |
| 执行方法/排障 | skill 或 runbook | PRD 产品定位 |
| 一次交付证据 | Delivery | memory 长期事实 |
| 长期稳定偏好 | memory 的短句 | Delivery 之外的大段复制 |
| 里程碑演变 | `PROJECT_EVOLUTION_REVIEW.md` | 每个微小修复都更新时间线 |

原则：总纲是一处事实源，其他文件链接它；代码和运行态证明实现；Delivery 记录一次执行，不承担长期规范。

### 第 10 步：生成一份高质量修改记录

一个可验证切片只生成一份 Delivery。纯只读检查、关键词搜索和多轮修正合并进同一份记录，不单独制造多份“收口”。

修改记录必须包含：

```text
Change ID：
目标和改前状态：
实际修改：文件 + 行为变化
关键决策：为什么这样做，明确没做什么
验证：命令 + 结果 + 证据
回归：覆盖了哪些旧能力
风险与遗留：
Git：每个仓库的 commit hash
部署：未部署 / 已部署 + smoke 证据
下一步：无，或一个明确的新 Goal
```

出现“请补充实际验证”“详细决策待补”“无”但实际有风险时，这份记录只能是 Draft。

### 第 11 步：Git 留痕

- 一个已验证切片对应一个可理解的 commit。
- commit 信息写用户结果或契约变化，不写“继续收口”“再次优化”。
- 跨仓库任务在两边 commit message 或 Delivery 中写同一个 Change ID。
- 提交前检查 `git diff --check`、目标文件 diff、构建/验收结果。
- 稳定改动验证后形成 Git 版本点；是否 push 服从钧钧当次要求和仓库状态。
- 不把不明数据库、密钥、缓存、构建垃圾或他人改动混入提交。

### 第 12 步：发布与结束

发布后再做一次真实入口 smoke。最终短报固定为：

```text
做了什么：
验证：
版本/同步到哪里：
结果与剩余风险：
下一步：
```

如果还有必须完成的同一 Goal 切片，就继续；如果只是新的增强想法，转为新 Goal，不用“下一步建议”假装当前任务未完成。

## 5. 修改记录分别放哪里

| 载体 | 唯一职责 | 不应承担 |
|---|---|---|
| Goal | 当前要得到的结果和边界 | 长期产品说明 |
| PRD | 产品方向、功能、路线 | 实际完成证据 |
| SSD | 系统语义、API、数据、UI 契约 | 聊天过程 |
| Backlog | 可执行项、状态、验收链接 | 大段设计说明 |
| Status | 某一时点的阶段快照 | 永久真相 |
| Code/runtime | 当前真实实现 | 产品意图解释 |
| Delivery | 一次切片的修改与验证证据 | 长期规范的唯一来源 |
| Git | 冻结版本、diff、回滚点 | 未定版想法 |
| Skill/runbook | 同类任务以后怎么执行 | 完整产品总纲 |
| Memory | 少量长期偏好和稳定边界 | PRD/SSD 复制品 |
| Sync latest | 最近交接摘要 | 全项目历史和唯一事实源 |
| Evolution Review | 里程碑和结构变化 | 日常流水账 |

`latest` 必须按域分开：`jieyi-latest`、`xiaoxue-latest`、`shared-latest`、`ops-latest`。全局 latest 只做索引，避免产品任务被模型或运维任务覆盖。

这是目标结构，目前尚未完成分槽。在 `VC-latest分槽` 验收前，仍按现行协议读取既有 Delivery 和 team sync，同时把“可能被其他任务覆盖”视为已知风险。

## 6. 推进规范

### 6.1 进度更新

执行者只在这些节点更新：

1. 开始：确认归属、基线和准备做什么。
2. 中途：完成一个真实切片，说明证据和下一切片。
3. 遇到边界：说明冲突、风险和需要钧钧决定什么。
4. 完成：按最终短报交付。

不要把每次打开文件、每个搜索命令、每个小修补都当成进度。

### 6.2 自动推进和停止边界

默认自动推进：

- 普通代码、接口适配、构建、只读数据排查。
- 文档、Delivery、低风险前后端修改。
- 用户级服务和页面验证。

必须暂停确认：

- 产品定位、核心语义或用户主路径取舍。
- 删除、重置、迁移或写入真实业务数据。
- 生产 cron、自动发布和对外消息。
- 密钥、Token、账号、模型、权限、网络和公网配置。
- 已有脏改与当前任务在同一文件严重冲突。

### 6.3 范围控制

- 当前 Goal 的验收未通过，不因为文档写得完整而结束。
- 新发现的小问题如果影响当前验收，纳入当前切片；不影响则进 Inbox。
- 连续三次修复仍无法通过同一验证时，停止猜测，汇总证据和尝试记录。
- 一次大规模文档同步之前，先确认总纲源，避免 20 个文件互相复制。

### 6.4 完成定义

一个功能只有同时满足以下条件才算完成：

- 目标行为已实现。
- 成功态、空态、错误态和必要兼容已检查。
- 约定的构建、API、用户路径和回归验证通过。
- 状态和必要事实源已更新，没有用文档替代代码证据。
- Delivery 没有占位文本。
- Git diff 可解释、可回滚；稳定阶段已有版本点。
- 真实运行入口已验证，或明确写“未部署”。

## 8. 立即执行的治理动作

以后每个新 Goal 从本手册执行。当前项目还需要独立完成以下治理切片，不能偷偷混入其他功能：

1. `VC-工作树分类`：把母仓库和独立小雪仓库的现有改动按来源、目标和风险分类。
2. `VC-小雪总纲版本化`：将当前文档总纲升级完成一致性检查、记录和独立版本点。
3. `VC-小雪runtime收口`：对当前前后端改动做 build、API、UI 和回归验收，并独立版本化。
4. `VC-Delivery门禁`：没有真实 `--verify` 时不得生成完成态 Delivery。
5. `VC-latest分槽`：按结衣、小雪、共享、运维拆分 latest；全局 latest 只做索引。
6. `VC-最小回归矩阵`：为结衣和小雪分别固定构建、API、页面与写回验证。

这六项是治理 Backlog，不代表当前产品功能未完成；它们解决的是“以后能不能稳定继续改”。
