## 1. 系统北极星（顶层定义）

**目标**：用户通过交易为手段实现财富自由。

**结构**：嵌套，非平等双 Agent。

```
财富自由（终极目标）
    ↑
  交易（当前手段）
    ↑
  小雪（交易专精辅助） ⊂ 结衣（人生统筹，含交易这一维）
```

- **结衣**：上位系统，统筹人生全部维度（含交易）。回答"怎么活"。
- **小雪**：结衣的子系统，专精交易决策辅助，数据回流结衣做人生层面复盘。回答"怎么打"。
- **关键推论**：不是双独立系统硬拼，是嵌套。所有功能归属按"服务于哪一层"判定。

---

## 2. 双 Agent 职责边界

| 维度 | 归属 | 回答什么 |
|----|------|---------|
| 交易技术复盘（这单为什么输/赢） | 小雪 | 怎么打 |
| 交易作为人生一维的整体反思（连续亏损对心态/纪律执行） | 结衣 | 怎么活 |
| TK 知识 = 交易决策知识 | 小雪 | 这单该怎么打 |
| 柱石库 = 人生原则（交易纪律是其子集） | 结衣 | 该怎么活 |

**知识体系**：维持分割不强行合并，通过引用关联。交易纪律是柱石库的子集（结衣），但引用小雪的 TK 决策知识。

## 3. 结衣：知行合一的数字分身（核心业务模型）

### 3.1 本质定义

结衣**不是仪表盘/执行器，是知行合一的数字分身**。

闭环：
```
知（学习/导入）→ 行（日程+执行）→ 思（反馈+提炼回流）→ 知（指导下次）→ ...
```

用户把知识喂进来 → 结衣把知识变成每日该做的事 → 用户做完反馈 → 提炼回流 → 循环精进。"知行合一做不到"是用户痛点，结衣就是来补这个的。

### 3.2 前端三页新体系（基于知行合一）

| 页面 | 旧定位 | 新定位 | 闭环角色 |
|------|--------|--------|---------|
| **知**（学习） | 系统页·堆原则 | 知识库 + 学习入口 | 知：导入 + 每日小学习 + 深度学习（留接口） |
| **行**（日程） | 行动台·待办 | 日程转化 + 执行 | 行：知识转化日程 + 随时可改 + 打卡 |
| **思**（反馈） | 回顾档案·发生了什么 | 记录反馈 + 提炼回流 | 思：所做所思所想 → 提炼成智慧回流知识库 |

柱石库/原则库归"知"——与导入知识并列（外部学 vs 自己提炼）。

### 3.3 "知"的两层含义

- **知道该怎么做**：外部知识 / 指导思想
- **知道你这个人**：个人记忆 / 对你的理解

拆解日程时两层一起用，才是真知行合一。

### 3.4 日程机制（"行"页）

**日程来源池（三类）**：
1. 知识库·外部知识/指导思想 → AI 拆解成可执行项
2. 知识库·对用户的记忆内容 → AI 拆解成针对性行动（结衣"知你全部"）
3. 临时加入（`user_add`）→ 已是行动，直接进日程

**规则**：
- AI 给建议（基于知识库 + 四象限重要紧急参考）+ 用户想加就加
- 日程是建议性非强制，核心治盲目松散（给方向感不死板）
- 数据结构需 `source` 标记：`ai_suggest | user_add`，随时可追加
- 四象限是参考性优先级提示，不是硬排序
- OpenMic 式深度学习：只留接口位，不实现（用户有想法未深挖）

**拆解链路**（已修正）：知识 → 拆解成可执行项 → 四象限参考 → 日程。先导入内容才能拆分。

### 3.5 "思"页记录机制（岁月式）

参考**岁月时间管理法**（微信小程序同名）：活动前后双记录 + 自动计时。

**活动记录来源**：
- 日程里有的直接点（关联 `schedule_id`）
- 没有的自己写（`schedule_id` 为空）

**岁月式机制**：声明下一项活动（承诺感）→ 自动计时 → 结束记体验/星级打分。

**两层粒度并存**：
- 细：活动记录流（按活动粒度，岁月式）
- 粗：每日总评（按天，原心情存档升级，含 `trade_ids`）

**行 vs 思打通**：行（日程）= 计划态；思（活动记录）= 实际态；通过 `schedule_id` 关联。

### 3.6 单次记录原则

每个事件只记一次，绝不重复记录。

- 交易只记在交易表（客观事实 + 情绪一句话），结衣"思"页自动 JOIN 展示，不让用户记两遍
- 结衣活动记录只手动记**非交易活动**，交易类活动记录是自动生成的视图

### 3.7 懒人版操作流（核心用户体验）

真实场景：早上扫日程 + 补临时项 → 白天有空边做边记（秒记）→ 晚上回顾写感悟。

| 操作 | 方式 |
|------|------|
| 情绪/状态 | 预设标签下拉选 + 进度条拖分（0-10） |
| 活动记录 | 点开始 → 点结束，体验拖星级/选标签，可留空 |
| 补临时日程 | 一句话即可 |
| 晚上回顾 | 看自动时间线，补一句感悟；系统自动生成今日总评草稿（含 trade_ids） |

**用户只做：选/拖/点/一句话感悟；系统做：汇总和关联。** 不打字填表。

## 4. 交易模块（核心新增闭环）

### 4.1 为何独立成表

之前缺失的核心闭环。小雪建表，数据回流结衣做人生层面复盘。

### 4.2 交易表字段（定稿）

| 字段 | 说明 |
|------|------|
| `trade_id` | 自动生成 |
| `date` | 自动记录 |
| `标的` | 关联赛事 |
| `调查` | 核心项 1，可加备注（字节级，可选） |
| `仓位` | 核心项 2，可加备注（字节级，可选） |
| `进场时机` | 核心项 3，可加备注（字节级，可选） |
| `结果盈亏` | 事后填 |
| `game` | 多电竞：`lol`（现有归位）/ `cs`（留空）/ `val`（留空） |

**符合懒人原则**：核心三项为主，备注可选，不堆砌字段。

**交互**：卡片式归档回顾，点卡片可编辑/加备注。

### 4.3 重叠切分原则（交易/心情/行动）

| 数据类型 | 归属表 | 存什么 |
|---------|--------|--------|
| 事实 | 交易表（小雪） | 客观事实 + pre_confidence + post_review（技术复盘） |
| 感受 | 心情表（结衣） | 情绪感受 |
| 计划 | 行动表（结衣） | 待办/日程 |

**关键**：
- 心情记录的 `had_trade` 升级为 `trade_ids` 数组，不重复存交易细节
- 交易记录不存情绪
- 查"交易时心态"通过 `date + trade_ids` JOIN，不重复存储

## 5. 长期互动式复盘

- **周期**：8 天 / 半月 / 月
- **载体**：飞书直接聊（结衣智能体），不用前端页面
- **原因**：对话式互动性强、随时聊、多轮挖深，前端表单死板
- **流程**：AI 按周期主动发起对话 → 多轮复盘 → 提炼成智慧条目 → 回流"知"页知识库
- **前端"思"页只展示提炼后的条目**，不承担对话本身

---

## 6. 飞书架构（现状，保持不动）

- 飞书连的是小雪、结衣**两个独立智能体**，已跑通，重构不改
- 前端两个端口独立（`:8880` 小雪 / `:3001` 结衣），松耦合好维护
- 双 Agent 互通走既有方式：文件系统 + 数据库共享，不走飞书机器人互通
- **教训**：曾建议"统一一个机器人"是越界，违反 100% 业务保留 + 减法原则，已收回

## 6.1 模型分工与切换可读性（架构约束）

**核心约束**：搭建期用强模型生成代码；日常运行走低成本模型控制成本。**具体用什么模型由部署时配置决定，代码不绑死**，唯一硬要求是保证后续切换模型的可读性。

| 场景 | 模型策略 | 说明 |
|------|---------|------|
| 搭建期（写代码/出稿） | 强模型，一次性 | 代码产出后运行期不依赖 |
| 日常运行·小雪 Agent | 低成本模型（部署配置） | 交易辅助、TK 查询、日报生成、交易记录分析 |
| 日常运行·结衣 Agent | 低成本模型（部署配置） | 日程拆解、活动记录、心情分析、每日总评草稿 |
| 日常运行·长期复盘 | 低成本模型（部署配置） | 飞书结衣智能体按周期发起复盘对话 |
| 采集入库 | 无模型 | 纯脚本（ScoreGG/TS/B站/公众号） |

**架构约束（保证切换可读性，必须遵守）**：

1. **Agent 调用层抽象成接口**：`agents/xiaoxue_agent.py` / `jieyi_agent.py` 内部封装 `LLMClient`，模型名/密钥/ baseURL 全走 config 环境变量，**不硬编码任何模型名**。切换模型只改 `.env` 不改代码。
   ```python
   # backend/app/config.py
   LLM_MODEL = os.getenv("LLM_MODEL")      # 不给默认值，强制部署时显式配置
   LLM_API_KEY = os.getenv("LLM_API_KEY")
   LLM_BASE_URL = os.getenv("LLM_BASE_URL")
   ```
   所有 Agent 代码通过 `LLMClient.chat()` 调用，不直接引用模型名。读代码时一眼能看出"这里调了 LLM，模型由配置决定"。

2. **Prompt 与代码分离**：各 AI 任务的 prompt 模板独立成文件（`agents/prompts/*.txt` 或常量），不散落在业务逻辑里。读代码时能快速定位"这个任务喂了什么 prompt"。

3. **前端不依赖模型**：前端代码纯展示与交互，AI 能力全走后端 API。日常换模型前端零改动。

4. **Prompt 工程面向低成本模型优化**：拆日程、活动记录、复盘对话等 AI 任务的 prompt 设计，需考虑低成本模型的能力边界——
   - 指令明确、步骤拆细、少零样本推理
   - 结构化输出优先（JSON），便于解析
   - 单轮任务为主，多轮对话控制上下文长度

5. **可读性优先**：LLMClient 接口、prompt 文件、config 配置项命名清晰，注释说明"为什么这样拆"。目标：换个 AI 来读代码，能立刻看懂模型怎么接、怎么换。

### 6.1.1 搭建期任务路由（按模型能力分层执行）

搭建期代码生成也按任务类型分配模型，控制单次执行复杂度，不一把梭：

| 任务类型 | 用什么模型 | 理由 |
|---------|-----------|------|
| 业务逻辑 / Agent 代码 / service 层 / Prompt 设计 | 强推理模型（如 qwen3.7-plus） | 需理解业务上下文、嵌套关系、知行思闭环、单次记录 JOIN 逻辑 |
| SQL / ORM 映射 / 建表 / 部署 yaml / 运维脚本 / 契约类型定义 | 结构化模型（如 deepseek-v4-flash） | 模板化、确定性强、不需深度推理 |

**执行顺序（三轮，有依赖）**：

```
Round 1（结构化模型·打地基）
  → 目录结构 + contracts 契约层 + DB 适配层 + 新表建表 + migrate_check + 采集模块迁移
  → 产出：后端骨架能跑、DB 能读原库、新表就绪、采集新旧对比

Round 2（强推理模型·写业务）  ← 依赖 Round 1 的契约和 DB
  → API 路由层 + Agent 代码 + 知行思 service 层 + Prompt 模板 + 单次记录 JOIN 逻辑
  → 产出：业务逻辑到位、双 Agent 承接、飞书现状不动

Round 3（结构化模型·收尾运维）  ← 依赖 Round 1+2 代码完成
  → 部署 yaml + docker-compose + systemd + cutover 清单 + 批量运维脚本
  → 产出：可部署、可切换、可运维
```

> 注：具体模型名（qwen3.7-plus / deepseek-v4-flash）是云端当前可用选择，不写死进代码，只在执行指令里标注。代码层面仍遵守 §6.1 的 LLMClient 接口化约束。

## 7. 技术栈选型

| 层 | 选型 | 理由 |
|----|------|------|
| 后端 | **Python 3.12 + FastAPI + SQLAlchemy 2.x** | 复用现有采集脚本/Agent 生态；SQLite/飞书/LLM Python 生态成熟；FastAPI 自动生成 OpenAPI 与前端类型契约对齐 |
| 数据库 | **SQLite（兼容读原库）** | 不替换数据库，新 ORM 映射现有表，禁止自动建表改结构 |
| 前端构建 | **Vite + React 18 + TypeScript** | 极简、启动快、符合私人自用；lazy routing 原生支持 |
| 前端路由 | **React Router v6** | lazy + import 模式 |
| 样式 | **Tailwind CSS** | 毛坯阶段快速搭结构，插槽化布局友好 |
| 包管理 | **pnpm** | 轻量、workspace 支持共享代码 |
| 契约层 | **contracts/ 单一数据源** | TS types + OpenAPI 双向对齐，前后端不各写一遍 |

## 8. 整体架构分层

```
┌─────────────────────────────────────────────────────────┐
│  前端（Vite + React）                                     │
│  xiaoxue-web (:8880)        jieyi-web (:3001)            │
│  ┌──────────────────┐       ┌──────────────────┐         │
│  │ 页面(空毛坯)      │       │ 知/行/思 三页      │         │
│  │ 插槽布局层        │       │ 插槽布局层        │         │
│  │ 配置化导航        │       │ 配置化导航        │         │
│  │ service 层(留接口) │       │ service 层(留接口) │         │
│  └────────┬─────────┘       └────────┬─────────┘         │
│           └──────────┬──────────────┘                    │
│              shared/ (types · api · config · components)  │
└──────────────────────┬───────────────────────────────────┘
                       │ HTTP (按 contracts 契约)
┌──────────────────────┴───────────────────────────────────┐
│  后端（FastAPI）                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ api 路由  │→│ services │→│repository│→│ DB 适配层 │   │
│  │(按契约)   │  │(业务逻辑) │  │(数据访问) │  │(读原SQLite)│   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │collectors│  │ agents   │  │ feishu   │                 │
│  │(8采集模块)│  │(双Agent)  │  │(收发)    │                 │
│  └──────────┘  └──────────┘  └──────────┘                 │
└─────────────────────────────────────────────────────────┘
                       │
                原有 SQLite 数据库（只读兼容，不动结构）
              + 新增表：trades（交易）/ activities（活动记录）
```

**分层职责**：
- **api 路由层**：只做请求分发 + DTO 转换，严格按 contracts 契约
- **services 层**：业务逻辑，承接双 Agent 职能
- **repository 层**：数据访问封装，隔离 SQL 细节
- **DB 适配层**：SQLAlchemy 映射原表，只读优先，写操作走 changelog 审计
- **collectors / agents / feishu**：独立模块，可被 cron/service 调用

## 10. 数据库适配层（兼容读原 SQLite）

**核心原则**：新 ORM 映射现有表结构，**只读优先，禁止自动建表/改表**。

```python
# backend/app/db/connection.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base
from app.config import settings

# 连接原库，不开 WAL 改写，echo 关闭
engine = create_engine(
    f"sqlite:///{settings.ORIGINAL_DB_PATH}",
    echo=False,
    connect_args={"check_same_thread": False},
)
SessionLocal = sessionmaker(bind=engine, autoflush=False, autocommit=False)
Base = declarative_base()
```

```python
# backend/app/db/models.py —— 映射资料里的 schema/tables.sql
# 关键：__table_args__ = {"extend_existing": True}，不自动建表
class Team(Base):
    __tablename__ = "teams"
    # 字段严格对应原表，类型对齐
    ...
```

### 10.1 新增表（本次重构新增）

| 表 | 归属 | 说明 |
|----|------|------|
| `trades` | 小雪 | 交易记录，字段见 §4.2，含 `game` 字段 |
| `activities` | 结衣 | 活动记录（岁月式），`schedule_id` 可空关联日程 |

> 新增表独立建，不动原表结构。读写分离：原表只读，新表可读写。

**读写策略**：
- **读原表**：repository 层封装查询，service 调用
- **写原表**：走 changelog 表审计（原库已有），记录 operation/old_values/new_values
- **写新表**：直接写，独立表无兼容包袱
- **迁移校验**：`scripts/migrate_check.py` 启动时跑一遍，确认 ORM 能正确读出原库各表行数与字段，**读不出来就停，绝不改原库**
