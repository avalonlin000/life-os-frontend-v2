# Life OS 项目演变与变更痛点复盘

> 复盘日期：2026-07-10
>
> 范围：`life-os-frontend-v2` 母仓库、当前独立小雪工作台 `/home/ubuntu/xiaoxue-web/`、产品文档、Delivery、team sync 和现行项目协议。
>
> 用途：解释项目为什么会反复修改，并给 `VIBE_CODING_PLAYBOOK.md` 提供事实依据。本文记录阶段性演变，不替代 PRD、SSD、Backlog 或 Git 历史。

## 1. 当前项目的真实形态

当前不是一个单体产品，而是“一层母工程、两个用户产品、多个运行仓库/服务”：

| 层级 | 当前事实 | 主要事实源 |
|---|---|---|
| 内部工程母项目 | Life OS，负责共享工程、总索引、交付规则和结衣前端 | `README.md`、`PROJECT_OWNERSHIP_INDEX.md` |
| 结衣知行合一 | 移动端优先的知、行、思、道闭环 | `packages/jieyi-web/`、结衣 PRD/SSD/Status |
| 小雪电竞人生 | 电竞交易辅助系统，LOL 是第一项目 | `/home/ubuntu/xiaoxue-web/`、小雪 PRD/SSD/Status |
| 历史小雪 React 包 | workspace 内的共享/历史包，不是当前主工作台 | `packages/xiaoxue-web/` |
| 运行与协作层 | Hermes bot、team sync、Delivery、skills、memory | `.hermes/`、`/home/ubuntu/.hermes/team/` |

这个结构是后期逐步形成的。早期“Life OS 是一个产品”“workspace 里的小雪就是主入口”等认识，后来都被修正，因此很多改动不是功能返工，而是项目边界持续定型带来的治理返工。

## 2. 证据概览

本次只读盘点得到以下事实：

- 母仓库共有 72 次提交：53 次 `docs`、12 次 `feat`、5 次 `fix`，其余为初始化和仓库治理。
- 盘点时 `.hermes/deliveries/` 有 100 份历史 Delivery（不含本复盘后续产生的交付记录），集中在 6 月 28 日至 7 月 6 日；其中 7 月 4 日 27 份、7 月 6 日 40 份。
- 盘点时 100 份历史 Delivery 和 `latest.md` 均命中默认占位提示，说明自动生成记录的数量很高，但关键决策、真实验证或同步摘要经常没有人工补全。
- 2026-07-10 母仓库存在 20 个已修改文件和 1 个新增文件，主要是小雪总纲升级；仅已跟踪文件的 diff 就有 1726 行新增、1085 行删除。
- 同时，独立小雪仓库存在 6 个已修改文件，diff 为 364 行新增、101 行删除，另有一个来源异常的未跟踪数据库文件。
- 两个主要仓库都处于 `main` 比远端 ahead 1 且工作树未清理的状态，跨仓库改动尚未形成同一个可回溯版本点。
- 根仓库没有统一的 `test` 或 `lint` 脚本；现有质量门主要依赖 build、API smoke、页面访问和人工检查。
- `.hermes/deliveries/latest.md` 最后停在 2026-07-06，但 Git 和工作树在 7 月 7 日至 10 日仍继续变化；“最新交付”已不能代表最新项目状态。
- `/home/ubuntu/.hermes/team/sync/project-latest.md` 会被不同类型任务覆盖，当前内容是 Codex/ccpocket 运维，不是 Life OS 产品最新状态。

这些数字不是为了否定已有工作。它们说明项目迭代速度很快，但记录系统没有跟上“两个产品、两个仓库、多类文档”的复杂度。

## 3. 项目演变时间线

### 阶段 A：前端 v2 建立（2026-06-19 至 06-20）

主要变化：

- 建立 React + TypeScript + Vite workspace。
- 结衣和小雪都先以 workspace 包存在。
- 补齐七页空态、Vite 代理和共享样式。

这一阶段解决的是“先能运行”。初始化提交同时纳入了大量依赖和构建产物，说明早期仓库边界、产物策略和版本卫生还没有稳定。

### 阶段 B：交付机制与产品拆分出现（2026-06-28 至 06-30）

主要变化：

- 建立 Delivery 生成和同步机制。
- 结衣四页主流程、统一复盘和 API 逐步成型。
- 小雪开始形成 LOL 横向基本面和 MSI 工作台。
- 项目从一个模糊的 Life OS，逐步拆为结衣、小雪和共享工程层。

这一阶段留下了 18 份 Delivery，但对应稳定 Git 版本较少，说明“过程记录”早于“版本快照”成熟。

### 阶段 C：蓝图、共享层与发布收口（2026-07-04）

主要变化：

- 共享 types/routes/services 开始拆分。
- 结衣内容样例、前端入口和轻交互进入页面。
- 部署、公网入口、release gate、视觉证据和仓库卫生被集中处理。
- 一天产生 27 份 Delivery 和 14 次提交。

该阶段把工程从“页面能跑”推进到“能部署、能验收、能交付”，但也形成了后续持续沿用的微批次风格：一次检查、一次 Delivery、一次状态更新，记录粒度开始小于真正的用户价值切片。

### 阶段 D：小雪日常入口产品化（2026-07-05）

主要变化：

- 独立 `/home/ubuntu/xiaoxue-web/` 成为实际小雪工作台。
- 今日内容、动作入口、依据层、错误态、动态日期、摘要条和产品边界提示陆续落地。
- 结衣 8881 后端运行基线被恢复和记录。
- 旧 TK 链路被兼容和降级。

这一阶段体现了典型的“先补入口，再补解释，再补错误态，再补日期”的连续修正。每次都有效，但如果在一个完整用户故事中设计和验收，可以减少多次触碰同一页面。

### 阶段 E：结衣从接口修复到完整闭环（2026-07-06）

主要变化：

- 修复 thinking cards、knowledge tags、knowledge split 等前后端响应不一致。
- 明确 fallback 不伪装真实材料、后端关闭不伪装成功、页面不主动填充 fake/mock。
- 补齐行动完成/重开/添加的写回核对，统一复盘和 daily review。
- 增加认知资产候选、10 至 14 天数据窗口、确定性模式识别、阻力信号与趋势总结。
- 同时统一 Goal/skill/Delivery、worker、完成度和产品归属口径。
- 一天产生 40 份 Delivery 和 40 次提交。

这是功能推进最快的一天，也是“记录碎片化”最明显的一天。大量接口查证、文档审计和功能实现被拆成独立交付，导致后续读者很难从记录数量直接判断真正发生了几个用户可感知的变化。

### 阶段 F：小雪交易辅助闭环升级（2026-07-07 至 07-10）

主要变化：

- 独立小雪工作台增加盘口工作区、操作闭环和赛前交易判断日报。
- 小雪定位从“LOL 横向基本面/日报工具”升级为“电竞交易辅助系统，LOL 是第一项目”。
- 三轴模型、能力层、交易前/交易时/交易后边界、阵容八步法和跨项目路线开始进入 PRD/SSD。
- 建立“小白主控、Codex 执行、GitHub 留痕、sync 闭环”的双项目协议。

当前仍处于这个阶段。文档总纲升级、独立小雪代码变化和 team 协议变化同时发生，但尚未被收成同一个明确的 Change ID、验收包和跨仓库版本点。

## 4. 反复出现的痛点与根因

| 优先级 | 痛点 | 反复表现 | 根因 | 新规则 |
|---|---|---|---|---|
| P0 | 项目入口和归属漂移 | Life OS 单产品、两个小雪目录、结衣 backend 位置多次澄清 | 项目结构先长出来，边界后补 | 每个 Goal 第一项必须先写“归属、主仓库、运行入口” |
| P0 | 事实源过多 | 同一定位同时写 README、PRD、SSD、BOT_GUIDE、skill、memory、sync、工作台 | 没有规定每种文件只负责一种事实 | 使用手册里的“事实源矩阵”是强制路由 |
| P0 | 完成口径不一致 | 方案完成、文档完成、接口实现、页面可用都曾被写成 Done | 缺少生命周期状态 | 只使用 Planned / In Progress / Implemented / Verified / Released |
| P0 | 跨仓库版本无法对齐 | 母仓库和独立小雪仓库同时 ahead、同时脏 | 没有跨仓库 Change ID 和提交对应表 | 跨仓库任务共享一个 Change ID，分别记录 commit hash |
| P1 | Delivery 量大、信息密度低 | 100 份记录，峰值一天 40 份，多数含占位文本 | 把每次检查当成交付，而不是把可验证价值切片当成交付 | 一个可验证切片只生成一份 Delivery；纯检查合并进同一验收包 |
| P1 | `latest` 不可靠 | Delivery latest 停在 7 月 6 日，team project-latest 被运维任务覆盖 | “最新”没有按产品和记录类型分区 | latest 必须按 `jieyi`、`xiaoxue`、`shared`、`ops` 分槽 |
| P1 | 文档同步造成大面积改写 | 一次定位变化触碰 20 多个文档 | 复制同一句定义，而不是引用总纲源 | 总纲只写一处，其他文档写摘要和链接 |
| P1 | 验证证据经常是模板 | Delivery 留有“请补充验证” | 先生成记录、后补证据的流程没有门禁 | 没有实际命令、结果和证据，不得标记 Verified/Released |
| P1 | 需求被拆得过细或过宽 | 一方面同一页面连续十次小改；另一方面一次改 20 份文档 | 缺少“用户行为 + 验收”的标准切片大小 | 一个切片覆盖一个用户行为或一个契约变化，并包含必要错误态 |
| P1 | 代码与文档不同步 | 文档说“仅总纲升级、未改代码”，同时 runtime 工作树已有大量代码变化 | 文档任务和代码任务没有共享状态 | Status 必须区分 Draft、Implemented、Verified、Released，并链接代码版本 |
| P2 | 自动化质量门不足 | 无统一 test/lint；依赖 build 和 smoke | 项目早期以快速可用为先 | 每个产品定义最小回归矩阵，逐步补测试，不用 build 冒充全量验证 |
| P2 | 用户说明和开发说明混在一起 | `PROJECT_USER_GUIDE` 同时写入口、启动、角色和开发边界 | 面向用户和面向修改者没有分册 | 用户使用看 User Guide；项目修改看 Vibe Coding Playbook |
| P2 | 脏工作树长期存在 | backend 曾因大量脏改无法提炼最小补丁 | 开始任务前没有强制基线和所有权声明 | 每个 Goal 开始前记录 dirty tree，未知改动不得覆盖或混交 |

## 5. 最关键的判断

### 5.1 主要问题不是缺文档，而是缺“单一变更单元”

当前已经有大量文档、Delivery 和同步文件。继续增加同类文件只会扩大漂移面。真正缺的是一个贯穿始终的 Change ID：

```text
Goal
-> 归属与基线
-> 可验证切片
-> 实现
-> 验证证据
-> 必要文档同步
-> Delivery
-> Git 版本点
-> Release/下一步
```

所有记录都应该围绕这个变更单元，而不是围绕“今天打开了哪个文件”。

### 5.2 文档不能用数量证明完成

产品总纲、PRD、SSD 的作用是定义目标和契约；代码、运行态和测试才证明实现；Delivery 只证明某次执行；Git 只冻结一个版本。任何一层都不能代替其他层。

### 5.3 下一阶段要从“收口很多次”变成“一个闭环收一次”

一个正常切片应同时考虑成功态、空态、错误态、必要兼容和验证。除非发现新的产品决策，否则同一用户行为不应被连续拆成多个“入口收口、提示收口、错误态收口、日期收口”。

## 6. 后续治理优先级

1. 先执行 `VIBE_CODING_PLAYBOOK.md`，以后新 Goal 全部使用 Change ID 和生命周期状态。
2. 当前未提交改动单独做一次“工作树分类”，不要与新功能继续叠加。
3. 将小雪文档总纲升级与小雪 runtime 代码分成两个已定义但有关联的切片，分别验证、分别提交，并在记录中互相链接。
4. Delivery 生成器增加门禁：未提供 `--verify` 时只允许 Draft，不得作为完成记录。
5. 将 `latest` 改为按产品/运维域分槽；全局 `project-latest` 只做索引，不再直接承载某个任务全文。
6. 为结衣和小雪分别补最小回归命令，逐步加入 test/lint；在此之前明确写出人工覆盖范围。
7. 每个里程碑只更新一次本文时间线，日常小改只进 Git 和 Delivery，不把演变文档写成流水账。
