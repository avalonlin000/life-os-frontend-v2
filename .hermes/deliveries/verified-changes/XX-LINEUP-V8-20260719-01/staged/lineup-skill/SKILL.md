# LOL 纯阵容完整分析 v8.0

## 目标

钧钧要看到完整分析过程，能亲自检查小雪有没有漏掉版本、场景或阵容断点。不要只给摘要，也不要恢复旧版的控制打分、0–100评分和队伍加权。

默认双方等经济、正常发育、同等操作水平。阵容只能给**倾向和条件**，不是比赛公式。

## 固定边界

- 只使用蓝红方十英雄、位置和比赛版本。
- 不读取队伍实力、选手状态、TS、三维、当前经济、装备差、赔率、赛果或社区观点。
- 允许读取正式通用版本知识和英雄机制；不得读取单队版本适配、队伍画像或交易记录。
- 不加载其他分析师 Skill，不运行控制评分或强度评分脚本。
- 不自动交易，不给仓位、跟单或买卖指令。
- 用户没给比赛版本时不要猜。按版本来源规则读取；仍无法确定就写 `版本 UNKNOWN`。

## 必读资源与版本动作

加载本 Skill 后，先读取 `references/version-and-24-scenarios.md`，再运行：

```bash
python3 /home/ubuntu/.agents/skills/lol-lineup-analysis/scripts/read_version_context.py --json
```

如果用户明确给了比赛版本，以用户版本为准，脚本内容只作通用框架参考。脚本返回 `conflict` 时必须展示冲突，只使用两个正式来源都支持的通用结论；不能静默选较新的那一份。

## 完整分析顺序

以下十步必须按顺序完整展示推导过程。UNKNOWN 也要展示，不能跳步。

### 1. 版本与输入确认

- 按上、野、中、下、辅列出双方十英雄和位置。
- 标出比赛版本、版本来源和来源状态。
- 位置摇摆或英雄不完整时先指出；不静默换位。

### 2. 版本契合

逐方判断体系、英雄环境、线权资源、装备伤害窗口、资源节奏和体系克制。每条必须引用正式版本结论或标 UNKNOWN，不能只写“更契合版本”。

### 3. 三路线权与野区联动

分别写上、中、下在 `0–6级 / 6级后 / 15–20分钟 / 20分钟后` 的机制倾向与反转点；再说明哪几路能先支援打野、第一轮河道和资源团。版本未知时降低置信度，不把机制倾向写成必然对线结果。

### 4. 阵容核心链

双方逐项展示：

`启动 → 第一轮承伤 → 主要输出 → 第二轮控制/追击 → 推塔或撤退`

同时回答前排是否稳定、主要输出是否安全、第一输出被断后有没有第二伤害点、赢团后能否连续转资源。

### 5. 关键控制链、反制链与替代赢法

- 每方写一条主要控制链和最容易断的位置。
- 最多列三个真正改变比赛的 Counter，说明覆盖范围和机会成本。
- 单个 Counter 只影响实际环节，不能宣布整套阵容失效。
- 主链受限后必须找独立替代路线；没有才算结构性问题。

### 6. 24场景定位

按 reference 的七项条件完整填写。输出一个主场景；不能唯一定位时列候选和缺失条件；再排除两个最接近但条件不成立的场景。场景用于描述走势，不重复算成额外优势。

### 7. 四阶段强弱

逐段输出 `0–10 / 10–20 / 20–30 / 30分钟后`。每段使用 `50/50、55/45、60/40、65/35` 粗略档位，并解释谁更舒服、为什么、如何转资源、是否发生强弱反转。比例是阵容相对倾向，不是真实胜率。

### 8. 比赛画像

- 节奏：快 / 正常 / 慢。
- 人头倾向：高 / 中 / 低；不猜具体数量。
- 走势：单向推进 / 运营拉扯 / 有来有回 / 胶着拖后。
- 说明场景定位如何支持这些标签，不能只贴标签。

### 9. 固定30分钟判断

固定以30:00基地被推掉为唯一分界，只选：

- 偏小：更倾向30:00前结束。
- 偏大：更倾向30:00后结束。
- 放弃：终结与拖延信号同时成立、版本来源冲突影响方向，或结论依赖单次高难操作。

必须展示支持偏小的证据、支持偏大的证据和最终取舍。不要输出30–35之类宽时间范围，后文不得反转方向。

### 10. 风险与观赛验证

- 列出两个最可能判断错的条件。
- 给两个“看到什么 → 说明什么”的现场信号：一个看关键技能与先后手，一个看站位、第二轮和资源转换。
- 不使用实时经济、补刀、装备和龙数重新预测；那些属于赛中综合判断。

## 输出要求

默认约1500–2500个中文字符，按十个标题完整输出。结论使用“更容易、相对舒服、倾向、如果……那么……”；避免“稳赢、必输、自动翻盘、拖住就赢、完全克制、接管比赛”等绝对表达。

发送前确认：版本来源可见、24场景有筛选过程、双方都有替代路线、四阶段与30分钟方向一致、没有混入队伍或交易资料。
