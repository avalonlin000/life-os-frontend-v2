## 一、当前架构适配评估

> 说明：本文件是多用户 SaaS 技术评估，不是当前产品归属标准。当前归属标准见 `docs/PROJECT_OWNERSHIP_INDEX.md`。Life OS 只作为内部工程母项目；用户侧产品已拆成 `结衣知行合一` 和 `小雪电竞人生`。

### 1.1 Hermes 团队架构现状

| 组件 | 当前形态 | 单用户 | 多用户需改动 |
|------|---------|--------|------------|
| **结衣（jieyi profile）** | Hermes profile + Feishu websocket | ✅ | ❌ 大改：每个用户需要独立 profile |
| **小雪（default profile）** | Hermes profile + FastAPI:8880 + SQLite | ✅ | 🔶 中等：DB 需加 user_id 隔离 |
| **Life OS 前端** | React + Vite，本地 dev server | ✅ | 🔶 中等：需加登录 + 用户路由 |
| **数据库** | SQLite 单文件 | ✅ | ❌ 大改：需换 PostgreSQL 或 SQLite 按用户分库 |

### 1.2 关键瓶颈

**瓶颈一：Hermes 的 profile 架构是单用户设计的**

Hermes 的 profile 机制（`~/.hermes/profiles/<name>/`）是为一个 Agent 服务一个用户设计的。每个 profile 有独立的：
- config.yaml（模型、工具、平台配置）
- session 数据库
- memories
- cron 定时任务
- skills

如果要支持多用户，每个用户需要一个独立 profile，或者对 Hermes 做二次开发支持多租户。

**瓶颈二：结衣的个性化记忆**

结衣目前通过 Hermes 的 memory 系统 + 知识库来记住钧钧的习惯、偏好、历史。在多用户场景下：
- memory 是按 profile 隔离的 → 天然支持不同用户
- 但知识库/数据文件是共用的 → 需要加 user_id 字段

**瓶颈三：前端无用户系统**

Life OS 前端目前是纯本地工具：
- 无登录/注册
- 无权限管理
- 无多用户路由
- API 请求直接指向本地后端

## 二、改造成本评估

### 2.1 最小可行多用户方案（估算）

| 改动项 | 工作量估计 | 复杂度 |
|-------|-----------|--------|
| 1. 数据库迁移：SQLite → PostgreSQL | 3-5天 | 🔶 中 |
| 2. 数据模型加 user_id | 1-2天 | ✅ 低 |
| 3. 前端加登录页 + JWT | 3-5天 | 🔶 中 |
| 4. 前端 API 层加 token 传递 | 1天 | ✅ 低 |
| 5. 后端（小雪）加用户注册/认证 | 2-3天 | 🔶 中 |
| 6. 结衣多用户隔离（memory 等） | 5-10天 | ❌ 高 |
| 7. 部署（域名、证书、预览链接等） | 2-3天 | 🔶 中 |
| **合计** | **17-29天** | — |

### 2.2 关键难点：结衣的多用户化

结衣是最难多用户化的部分，因为：
1. Hermes 当前主要按 profile 隔离使用，不是现成 SaaS 多租户产品
2. 每个用户需要自己的记忆空间
3. Feishu bot 目前只绑定了钧钧的群聊

**可能的方案：**
- 🅰 每个用户建一个 Hermes profile（笨但可行）
- 🅱 自己写一个轻量 multi-agent 路由层（更灵活，但开发工作量大）
- 🅲 放弃结衣的多用户版本，只卖方法论（电子书/课程），结衣只服务钧钧自己（推荐）

### 2.3 推荐路线

```
Phase 1（现在）：纯自用 → 不用改
Phase 2（短期）：卖方法论 → 只出文档/课程，不需要多用户
Phase 3（中期）：AI班主任服务 → 先做1v1模式（每个用户一个独立实例）
Phase 4（长期）：SaaS化 → 再考虑多租户架构
```

## 三、为什么不建议现在做多用户？

1. **产品还没定型** — 自己都还没用顺就做多用户，会浪费大量时间在架构上
2. **核心价值是结衣的方法论** — 不是代码
3. **商业模式未验证** — 不知道有没有人愿意付费
4. **钧钧的时间最值钱** — 完善体系 > 重构架构

**建议：**
- 继续打磨 Life OS 自用体系
- 把方法论沉淀成可输出的内容（课程/电子书）
- 有真实用户需求时再考虑多用户架构

## 四、如果未来要做多用户，推荐技术栈

| 组件 | 推荐 | 原因 |
|------|------|------|
| 数据库 | PostgreSQL | 原生多租户支持，row-level security |
| 认证 | Auth0 / Supabase Auth | 开箱即用，省去开发 |
| API 认证 | JWT | 简单，前端易集成 |
| 部署 | Docker + VPS / Railway | 成本低，易扩展 |
| 前端框架 | Next.js / Remix | SSR + 路由更成熟 |
| 每个用户一个 Agent | Hermes profile 动态创建 | 利用现有能力 |
